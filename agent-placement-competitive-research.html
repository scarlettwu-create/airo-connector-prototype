Agent Placement — Competitive Research

Context. Research compiled to inform AIRO product decisions on where and how to surface AI agents in the UI. Covers how enterprise and productivity platforms handle agent entry points, interaction surfaces, and the visual distinction between an autonomous agent and a conversational assistant.

Products covered. Notion, Slack, ServiceNow, Salesforce (Agentforce), Microsoft 365 Copilot, Linear

Scope. Where in each product's UI the agent appears (entry point, surface, panel type), how users invoke it, whether it is context-triggered or user-initiated, and how the agent interaction differs from simpler inline AI features.

Part 1 — Agent placement per product

1.1 Notion

Notion separates AI into three tiers: inline editing assist, Q&A search, and agents. Each has a distinct entry point.

Inline editor AI is triggered by pressing the spacebar on an empty line (opens an inline prompt bar in the page body), using /AI Block (inserts a persistent AI-generated block), or highlighting text (surfaces a floating toolbar with options like rewrite, shorten, or translate). Output appears inline as a new block below the cursor. There is no modal or panel for this tier.

Notion AI Q&A lives in two places: the Search entry in the left sidebar (opens a full-page chat view) and a sparkle icon in the bottom-right corner of any page (opens a side panel alongside the current page). The keyboard shortcut Shift+Cmd/Ctrl+J opens the Q&A side panel without leaving the page. Responses stream in with source links. The side panel variant is the closer analog to what most platforms call an "agent chat" surface.

Notion Agents (launched September 2025) are manually invoked for now. Each agent has a profile page where the user sets instructions, data sources, and output location. Agents can create pages, populate databases, and pull from connected tools (Slack, Google Drive, GitHub) for up to 20+ minutes autonomously. Custom Agents (February 2026) run on a schedule without manual prompting; every run is logged and reversible.

The design decision worth noting: there is no standalone "AI" tab in the left nav. AI is contextual. The bottom-right sparkle icon is the closest thing to a persistent global entry point, and it only opens the side panel, not the agents product.

1.2 Slack

Slack's approach is structurally different from every other platform here: agents are not panels, they are teammates.

Slack AI (the built-in assistant) is invoked via the "sunroof" -- a persistent icon in the top-right corner of the header bar. Clicking it opens a right-side split-plane panel where you interact with Slack AI or any enabled agent. This is the only product in this research where AI has a dedicated spot in the global header (not a sidebar icon, not a bottom button). Thread summaries appear inside the thread via a "Summarize" button at the top; channel summaries are triggered from the three-dot channel menu. Huddle transcriptions and summaries are delivered automatically after a huddle ends.

Third-party agents are installed from the Slack Marketplace and then surface as bot users in DMs and channels. Once installed, they appear in the sunroof panel so users can browse and add them. Agents can be @-mentioned in any channel or DM and respond in-thread like a team member. An Agentforce Hub (launched early 2026) acts as a directory inside Slack where users browse available agents, start new conversations, and review prior interactions. Each agent also gets its own DM-style conversation view with consistent UI including streaming text and suggested prompts.

Agentforce agents in Slack use the same DM/panel system -- there is no separate Einstein-branded button. Users @-mention the specific Agentforce agent (Sales, IT Service, HR Service) in a DM or channel.

The design bet: treating agents as teammates (addressable by @-mention, appearing in the sidebar like coworkers) rather than as utility panels makes them feel ambient and persistent. The tradeoff is discoverability -- users may not know which agents are installed or what they can do without going to the Agentforce Hub.

1.3 ServiceNow

ServiceNow has the most segmented placement strategy in this group: the surface changes entirely depending on whether you are an end user (employee), an IT technician, or an admin.

For employees, Now Assist appears as a floating action button (FAB) with a sparkle icon on the Employee Center portal page. Clicking it opens a full conversational window with multi-turn dialogue, prior conversation history, and a live agent handoff option. This replaces the older floating chat bubble.

For IT technicians in Agent Workspace, the same sparkle icon lives in the top-right corner of the platform header. Clicking it opens a right-side slide-out panel that auto-populates with a contextual incident summary and suggested next steps when the agent opens a ticket. They see AI that is proactive and scoped to the ticket in front of them. Additionally, a sparkle context menu icon appears inside text fields (work notes, email replies) anywhere AI writing assist is enabled -- clicking it opens an inline popover with writing options.

Virtual Agent (the older rule-based chatbot) uses a floating chat widget on Service Portal and Employee Center. Now Assist can layer GenAI responses inside Virtual Agent conversations, but the widget itself stays in place. These are separate products sharing one channel.

AI Agent Studio is the admin-facing builder where multi-step autonomous agents are configured. End users never see this UI; they see agentic workflows surface through the right-side Now Assist panel when a task requires multi-step automation. The panel can show the agent's reasoning and pause to ask a clarifying question.

Otto (announced late 2025) is a unified conversational AI layer across the platform, accessible via chat, voice, mobile, and web. It routes requests to the right underlying agent. Think of it as the meta-layer above individual Now Assist skills.

The pattern: ServiceNow has three distinct UI surfaces (floating FAB for employees, header panel for technicians, inline context menu for field-level assist) that share the same AI backend. The entry point changes based on the user's role and context.

1.4 Salesforce (Agentforce)

Salesforce draws a hard distinction between a conversational assistant (Agentforce Default) and autonomous agents that operate without a human in the loop. That distinction shows up directly in the UI.

Agentforce Default (the direct successor to Einstein Copilot) is a right-side chat panel that overlays the current Lightning page without navigating away. It is user-initiated: the rep clicks an icon in the Lightning nav bar (utility bar or bottom-left area depending on org configuration) to open it. In Service Cloud's Agent Workspace, this panel is always present on the right side alongside the case feed and proactively surfaces contextual suggestions -- warranty checks, suggested next steps -- when a case record loads.

Autonomous Agentforce agents are largely invisible to internal Salesforce users while running. They execute tasks, respond to customers (via messaging channels or portals), and log outcomes. Internal users see results in record updates and conversation history transcripts, not in an interactive chat panel. This is a deliberate design decision: the agent takes action, the human reviews outputs.

Agent Shortcuts (Summer 2025) are a third entry point: admins can add AI-powered action buttons directly into Lightning record page layouts. A Quick Action button triggers an agent task without opening the chat panel at all. This collapses the multi-step "open panel, type request, wait" flow into a single click for common workflows.

For external users in Experience Cloud portals, Agentforce deploys as a chat widget (messenger-style, embedded via Omni-Channel). It can also be embedded on external websites via a script snippet.

Agentforce Builder (admin tool) has a three-column layout: a left navigator/explorer pane, a center canvas for editing agent instructions and topics, and a right preview panel where admins test live conversations and view the Plan Tracer to inspect agent reasoning steps.

1.5 Microsoft 365 Copilot

Microsoft distributes Copilot across every surface in M365 rather than centering it in one place. The entry point changes by app.

In Teams, Copilot appears as a persistent icon in the left rail app bar. Inside chats, an "Open Copilot" button sits in the upper-right corner of the thread. During meetings, Copilot is an in-meeting overlay panel; post-meeting, it surfaces in the recap tab to summarize transcripts and generate action items.

In Outlook, Copilot lives as a side panel accessible from the toolbar. For email compose, a Copilot icon in the compose toolbar offers draft assist. In the reading pane, it summarizes long threads. A 2026 rollout is adding agentic actions (scheduling follow-ups, managing inbox priorities) from within this panel.

M365 Chat (formerly BizChat) is the primary cross-app entry point: it is a dedicated Teams app in the left rail and is accessible at microsoft365.com/chat. It handles queries that span emails, files, meetings, and calendar. On Windows 11, Win+C or the dedicated Copilot hardware key on newer keyboards opens a floating prompt box. Custom agents are @-mentioned inside M365 Chat, not surfaced as separate DMs.

Custom Copilot Agents are built in Copilot Studio and published to the Teams + M365 Copilot channel. Admins review and approve them through the Microsoft 365 Admin Center. Once approved, they appear as addressable entries inside M365 Chat.

The pattern across M365: Copilot is never in the center of the screen. It is always a panel, overlay, or secondary column that sits beside the content the user is already working on.

1.6 Linear

Linear started with inline AI writing assist and is now building toward a full conversational agent.

Inline AI is triggered contextually. In the issue editor, a Copilot-style toolbar button opens a chat overlay for spec drafting or writing help. There is no persistent side panel -- it is modal/overlay-based and dismissed after use.

Triage Intelligence runs automatically on incoming issues. Suggested properties (team, assignee, labels, priority) surface directly on the issue card or list view without user invocation. As of September 2025, auto-apply can be enabled so suggestions are applied without manual review at all. This is the closest Linear gets to an ambient agent that acts before the user asks.

Code Intelligence connects Linear to GitHub: it auto-attaches relevant files and functions to issues, auto-drafts PR descriptions from issue specs, and surfaces issue context on new pull requests. PR numbers appear inline in list and board views. No UI panel needed -- the linkage is automatic.

Linear Agent (launched March 2026) is the full conversational interface. It opens as a maximized overlay from a toolbar chat button -- described by Linear as "a natural extension of the toolbar rather than a separate page." Available in web, desktop, mobile, and as a plugin for Slack, Teams, and Zendesk. Users can save repeated agent conversations as reusable "Skills" for recurring workflows (e.g., draft issues from meeting notes, generate project catch-ups).

The Linear approach is the most conservative in this group for UI surface: there is no persistent panel, no left-rail AI entry, and no role-based surface segmentation. AI is triggered contextually from wherever the user already is.

Part 2 — Cross-cutting patterns

Pattern 1: The sparkle icon is the universal shorthand for "AI is here"

Every product in this research uses a sparkle/star icon as the visual cue that AI is available: Notion's bottom-right sparkle, ServiceNow's header sparkle and field-level sparkle, Salesforce's Copilot icon (which is a constellation variant). Slack uses the "sunroof" button (its own branded icon) but it occupies the same header-right position. This has reached the point of a de facto convention. Users moving between enterprise products already know that the sparkle = AI entry point.

Pattern 2: The right-side panel is the dominant interaction surface

With the exception of Slack (which treats agents as bot users) and Linear (which uses a maximized overlay), every product here puts the agent in a right-side slide-out or persistent panel that appears beside -- but does not replace -- the content the user is working on. ServiceNow (right-side panel in Agent Workspace), Salesforce (right-side overlay panel in Lightning), Microsoft 365 (side panel in every app). The shared design assumption: the agent is a collaborator on the work you are doing, not the primary task.

Pattern 3: Context-triggered agents are differentiated from user-initiated assistants

Three products in this group have moved to proactive, context-triggered AI: - ServiceNow's Now Assist panel auto-populates with a ticket summary and suggested steps the moment an IT technician opens an incident. - Salesforce's Service Rep Assistant proactively surfaces next-step recommendations when a case loads. - Linear's Triage Intelligence applies labels and routing suggestions automatically before the user touches the issue.

The user-initiated pattern (click the icon, type a request) is the default for general assistants (Notion Q&A, Slack AI, Microsoft Copilot). Context-triggered AI is appearing in narrower, high-frequency workflows where the latency of "open panel, type request" is friction enough to matter.

Pattern 4: Autonomous agents are mostly invisible to internal users while running

Salesforce's autonomous agents don't surface a chat panel to the rep while executing -- the rep sees record updates and transcripts after the fact. ServiceNow's agentic workflows appear in the Now Assist panel only when they need to pause and ask a clarifying question. Notion agents run in the background and are reviewed by looking at their profile page logs.

This is a meaningful divergence from the coding agent space (Devin, Replit, Cursor) where the whole product is built around watching the agent work. Enterprise SaaS platforms are treating agent output as a record system concern (show me what happened) rather than a live observation concern (watch it happen). Whether that is a product philosophy or a technical constraint is unclear, but the pattern is consistent.

Pattern 5: There is no solved model for distinguishing "assistant" from "agent" in a single UI

Salesforce has made the sharpest distinction: Agentforce Default (conversational, human-in-loop) shows up as an interactive chat panel; autonomous agents are invisible while running and surface only in logs. Every other product blurs the line. Notion's Q&A side panel and Notion Agents are both reached via the same sparkle icon but have very different autonomy levels. ServiceNow's Now Assist panel handles both short inline writing assist and multi-step agentic workflows without clearly flagging which mode you are in.

Users currently have no visual vocabulary for "this thing is acting autonomously right now" versus "this thing is waiting for my next message." That gap creates both a design opportunity and a trust/oversight risk.

Pattern 6: Agent discoverability is an unsolved problem

Slack's Agentforce Hub and Teams' @-mention model both assume users know what agents exist. Salesforce's Agent Shortcuts put common agent actions directly in record page layouts. ServiceNow's Otto tries to be the meta-router so users don't have to know which underlying agent to invoke. None of these are fully satisfying: Slack's hub requires the user to go looking, Teams @-mentions require knowing the agent's name, Agent Shortcuts require admin configuration per use case. The products that have invested in proactive/contextual surfacing (ServiceNow, Salesforce Service Console) seem to have the cleaner UX but at the cost of very narrow scope.

Part 3 — Design implications

Cross-cutting pattern What this could mean for AIRO
Sparkle icon is universal shorthand If AIRO has a persistent entry point in the UI, the sparkle/constellation iconography is already legible to enterprise users and probably the right choice rather than inventing a new symbol
Right-side panel is the dominant surface A right-side panel that appears beside the user's current context (a recipe, a build, a dashboard) is the most conventional choice. Worth deciding early whether this panel is always visible or slide-out-on-demand
Context-triggered vs. user-initiated If AIRO can detect that a user has just opened a specific type of resource or triggered a specific event, a proactively populated panel (pre-summarized, with suggested next steps) could reduce the "what do I ask?" cold start. Most platforms only do this in narrow, high-frequency workflows
Autonomous agents are invisible while running For AIRO, deciding whether users should watch the agent work (the Devin/Cursor model) or just see the output (the Salesforce model) is a foundational product question. Enterprise SaaS is converging on the latter
No solved model for assistant vs. agent in one UI If AIRO has both a conversational mode and an autonomous mode, the UI surface for each should probably be visually distinct. Using the same panel for both creates confusion about what the system is doing and whether it needs a response
Agent discoverability is unsolved Pre-populating the panel with suggested actions when users land on a relevant context (instead of leaving an empty text box) is more discoverable than relying on users knowing what to type. Salesforce's Agent Shortcuts and ServiceNow's proactive ticket summary are the best examples of this pattern

The biggest gap across all six products

Every platform has a solution for where the agent lives. None of them have a clean solution for communicating what the agent is doing or has done in a way that builds user trust over time. Enterprise platforms surface results in logs and record updates; developer tools surface everything but create noise. AIRO probably lands in between: users who care about recipes, builds, and automation flows will want to see meaningful steps (not every token), understand when and why the agent made a decision, and have a clear trail without sifting through a raw activity feed.
