Loading Status / Progress Indicators in AI Coding Tools — Competitive Research

Context. Research into how AI coding and app-building agents show the user what's happening during the wait between sending a prompt and getting a result. For the AIRO loading/progress experience while the agent plans and builds automations.

Products covered. Cursor, GitHub Copilot (VS Code agent mode), Claude Code, Replit Agent, Lovable, Devin.

Scope. What the UI shows while the agent is working (spinners, streaming reasoning, tool-call logs, plan/todo checklists, elapsed time, token and compute meters), and how a user can tell the agent is making real progress versus stuck.

Part 1 — Loading-status behavior per product

1.1 Cursor

While the agent runs, chat shows a self-updating to-do checklist plus streaming tool calls and diffs, with top-level status labels like "Planning Next Moves" and "Generating", and you can queue follow-up messages that fire "at the next ideal time, usually after a tool call" (Cursor 1.2, Cursor 1.4, Plan Mode).

1.2 GitHub Copilot (VS Code agent mode)

"Every tool invocation is transparently displayed in the UI", terminal commands surface inline for approval, the agent self-heals on build/lint/test output, and with Planning on it writes a user-facing Markdown plan that "updates visibly in the editor" (agent mode intro, Visual Studio agent mode, chat agent mode reference).

1.3 Claude Code

The CLI shows an animated spinner with rotating status verbs, a live elapsed-time counter, a running token count, and an "esc to interrupt" hint, with a "Thinking for Ns" counter that warms the spinner to amber after 10 seconds to signal it's still working (changelog, entries 2.1.152 and 2.1.154).

1.4 Replit Agent

The durable progress surface is the checkpoint timeline (each checkpoint has an AI-generated description, timestamp, and change scope, viewable in the Agent tab, Git pane, and History view), and Agent 3 adds a computer-use testing replay plus mobile live monitoring with progress bars (checkpoints and rollbacks, updates index).

1.5 Lovable

Chat shows a condensed agent activity card that groups tool calls and actions with a details view of the full action timeline, contextual tips appear while processing, and the live preview now refreshes once per agent update instead of flickering on every change (changelog: Feb 5, Apr 24, and May 21, 2026).

1.6 Devin

The session UI centers on a Progress tab that unifies shell commands, code edits, and browser activity in one clickable step log, with a long-session warning that fires around 2.5 hours or 10 ACUs, and a post-run Session Insights view showing a color-coded timeline plus ACU compute consumed (session tools, session insights).

Part 2 — User pain points (themed insights)

Five themes emerged from mining the Cursor forum, the anthropics/claude-code and microsoft/vscode issue trackers, Replit Discourse, Vercel Community, and product-review writeups. The complaint lives on vendor forums and GitHub far more than on Reddit or HN. The throughline: a spinner that keeps animating is not the same as evidence the agent is doing something, and users have learned not to trust the difference.

Theme 1: "Is it stuck, or still working?"

The single most repeated complaint, and it shows up in every product. A bare spinner or a top-level status label tells the user that something might be happening, but gives them no way to tell a long-but-healthy run apart from a hung one.

  • Cursor: "a 'Planning Next Moves' or 'Generating…' entry cannot be expanded, and is a complete and total BLACK BOX." — jrista, Cursor forum, Jan 2026
  • Claude Code: "The spinner is spinning but not any other action happens. When I stop it and say to it, continue, it continue without problem though." — aliovou, claude-code #41683, Mar 2026
  • Replit Agent: "they start working then don't do anything, it hangs and gives no feedback." — jefftj86, Replit Discourse, Sep 2025
  • Devin: "Sometimes it can be appear to be idle for a few minutes." — The Ground Truth, Devin first impressions, 2024

For AIRO: the loading state needs to answer "is this normal or is it broken" without the user having to stop the run to find out. The act of stopping to check (which several users do) is itself the failure.

Theme 2: The indicator lies, it animates while nothing happens, and still bills you

The sharpest version of the problem. The animation is the user's proof of life, so when it keeps moving while the actual work has stalled, the indicator manufactures false confidence. In credit-based products that false confidence costs real money, because the meter keeps charging for the dead time.

  • Claude Code: "The spinner metadata shows a long elapsed time (1m56s) but minimal thinking time (1s), suggesting the process is stuck waiting on an API response that never arrives." — jlmalone, claude-code #24688, Feb 2026
  • Copilot: stuck in a "Thinking" state for 25+ minutes "showing that it's reading files, but never suggesting or applying code edits." — globalvideos272-lab, community discussion #190804, Mar 2026
  • Replit Agent: "I continue to get x seconds of work performed - and charged approx 25 cents each time - but literally no work is happening." — jacobmortimer1, Replit Discourse, Oct 2025
  • Replit Agent: "the agent decided on action but did not implement anything. It created a checkpoint and charged my account." — joeschmoeno, same thread, Oct 2025

For AIRO: if the progress indicator is driven by a timer or a looping animation, it will look alive during a stall by construction. Deriving it from real signals (an active tool call, tokens actually streaming, a step state changing) is what lets it show "stalled" honestly. This matters more in any usage-metered context, where a loader that lies is also a meter that overcharges.

Theme 3: It goes silent mid-task, and you have to poke it to revive it

Distinct from a stall at the start. Here the agent streams partway, then stops, with no signal that it paused or why. Users learn a workaround: type "continue" to wake it, which means the product has trained them not to trust completion.

  • Cursor: "Agents often stops generating output halfway through a response" and "becomes completely unresponsive... happening roughly every one or two messages." — Z_W, Cursor forum, Jan 2026
  • Cursor: "cursor will silently pause on any action." — andrew0517, Cursor forum, Apr 2026
  • Copilot: "the terminal hangs with the message: Running command in terminal… No further progress is made, and the process does not complete." — sly01, vscode-copilot-release #4814, Feb 2025
  • v0: "v0 constantly freezes, stops generating code, and returns incomplete responses." — blitzmic, Vercel Community, May 2025

For AIRO: a partial-then-silent state is worse than an outright error because nothing marks it as a problem. Worth deciding what the UI does when streaming stops but the run hasn't reported completion, rather than leaving the last frame frozen on screen.

Theme 4: No ETA, no ceiling, no sense of how far along

Users can sit for minutes to hours with no indication of how long the run will take or how much is left. The few products that surface a bound stand out for it.

  • Copilot: "sometimes it just gets stuck at 'Working....' and it does not timeout, can sit there for 10 minutes." — bakman2, vscode #251475, Jun 2025
  • Cursor: "get stuck for very long time (even after a few hours still no progress)." — chi_hay_Yip, Cursor forum, Jan 2026
  • Replit Agent: "replit agent is searching for past 13 min and its set on low autonomy tho and i tried with max also." — TheCodingLord, Replit Discourse, Sep 2025
  • Devin, as a counter-example, does surface a bound: "A warning appears when the session has been going on for a long time (at around 2.5 hours or 10 ACUs)." — The Ground Truth, Devin first impressions, 2024

A true percent-done bar is hard for an agent because the work isn't known up front. Devin's move is the realistic version: surface a compute meter (ACUs) and a long-session warning so the user has some ceiling to reason against, even without a precise ETA.

For AIRO: worth deciding what bound the UI can honestly show. A step count against a known plan, a compute or cost meter, or a "this is taking longer than usual" threshold all beat an unbounded spinner.

Theme 5: When the loader freezes, it can take your work with it

The most damaging version. The wait state doesn't just go opaque, it locks up and loses the session, so the cost of an ambiguous loader is not just confusion but lost work.

  • Lovable: "I've spent over 20 credits trying to fix a code that results in my app being frozen on a 'loading page'." — Vibe Coding With Fred, retrospective, 2025
  • v0: "It will be in the middle of writing a file, and just completely freeze, and it will infinitely say generating." — Salvinoto, Vercel Community, Feb 2025
  • v0: "seems when trying to load projects they infinitely load." — Mash, Vercel Community, Oct 2025
  • v0: "V0 suddenly froze and I lost the entire chat and all my progress." — Baytech PH, Vercel Community, Jul 2025

For AIRO: the loading state and session durability are linked. If a frozen loader can lose in-progress work, the progress UI needs a recovery path (resume, or at least preserve what was built so far), not just a better animation.

Part 3 — AIRO design implications

Competitor pain point What this could mean for AIRO
Bare spinner / "Generating" label is a "black box" users can't read (Theme 1) Worth showing the current action in plain language (what tool is running, what step in the plan), not just an animated state label
Indicator animates while work has stalled, and still bills (Theme 2) Worth deriving the progress signal from real activity (active tool call, streaming tokens, step transitions) so it can show "stalled" honestly rather than always looking alive
Agent goes silent mid-task with no marker (Theme 3) Worth defining what the UI does when streaming stops without a completion signal, so a partial-then-silent run is visibly flagged, not left frozen
No ETA or ceiling on long runs (Theme 4) Worth surfacing some honest bound: step count against the plan, a compute or cost meter (Devin's ACU model), or a "longer than usual" threshold
Frozen loader loses the whole session (Theme 5) Worth linking the loading state to session durability so a hang has a resume or preserve path, not just a spinner
Tool calls shown transparently (Cursor, Copilot, Devin) Worth treating the activity log as the primary progress surface, with the spinner as secondary
Plan/todo checklist that updates live (Cursor, Copilot, Lovable) Worth showing the plan up front and checking off steps as they complete, which doubles as a natural progress bar for work whose length isn't known
Checkpoint timeline with descriptions and timestamps (Replit, Devin) Worth making each completed unit of work a legible, timestamped entry, so "what has it done so far" is answerable mid-run
Long-session warning + compute meter (Devin) Worth a soft ceiling that warns before a run gets expensive or runaway, especially in a metered context

The biggest gap surfaced by this research

Across all six products, the loading indicator is built as decoration, a spinner or animation on a timer, rather than as a truthful readout of whether work is actually happening. That design choice is the root of the most-repeated and most-damaging complaints: the indicator keeps moving during a stall (Theme 2), so users can't tell stuck from working (Theme 1), and in metered products the meter keeps charging for dead time. The open question for AIRO is whether the progress indicator should be derived from real signals (an active tool call, tokens streaming, a step state changing) so it can honestly show "stalled," instead of a time-based animation that always looks alive. This is a higher-stakes decision for AIRO than for an IDE assistant, because AIRO builds automations that may have side effects in connected systems, so "is it still working" is also "should I intervene before it does something."

Part 4 — Framework: showing the work as the progress signal

Sources. Shape of AI's Stream of Thought pattern; Nielsen Norman Group on progress indicators and response-time limits; Interactive Reasoning: Visualizing and Controlling Chain-of-Thought Reasoning in LLMs, arXiv 2506.23678 (ACM IUI 2025).

The classic UX rules apply to agents almost unchanged. NN/G's thresholds say anything past about 10 seconds needs determinate progress plus a way to interrupt, because an indeterminate spinner leaves users unable to tell whether the system is working or stopped, so they abandon the task. Agent runs almost always blow past 10 seconds, which puts them squarely in the "must show real progress and an interrupt" regime. Most of the products here ship the interrupt (esc, stop) but not the real progress.

What's new for agents is what the "progress" content should be. Shape of AI's Stream of Thought pattern names it: the progress surface is the agent's plan and work log itself, with each step treated as an explicit state (queued, running, waiting for approval, error, completed). It also argues for showing an editable plan before execution so users can correct course early. That state model is directly usable design vocabulary, and it reframes the spinner as the fallback, not the main event.

The IUI 2025 study adds the evidence that this is not just trust theater. In a 16-person user study, surfacing structured reasoning let participants catch and interrupt erroneous generations early and steer the model mid-run. The payoff of exposing intermediate work is concrete: earlier error-catching and mid-run intervention, not just reassurance.

Constraints worth understanding

  • A true percent-done bar is usually not achievable for an agent, since the total work isn't known up front. A step-against-plan count or a compute meter is the realistic substitute.
  • Showing the full reasoning stream can be verbose and overwhelming; the pattern calls for compact summaries and collapsible detail, not a raw token dump (Cursor's Compact mode and Lovable's grouped activity cards are attempts at this).
  • A live work log only helps if it reflects reality. If the displayed steps are model narration rather than the actual tool calls, it reintroduces the Theme 2 problem (the display says one thing, the system did another).
  • Plan-before-execution adds an upfront step that trades a little speed for steerability, which is a deliberate product call, not a free win.
