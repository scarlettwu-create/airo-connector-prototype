Coding, testing, and version control: how AI products wire the flow — Competitive Research

Context. Research for the AIRO SDK editor (PMO-2980) and recipe building, looking at the relationship between three steps that used to be separate tools and are now collapsing into one agent loop: writing code, testing it, and committing it to version control. This is a research-stage artifact, before AIRO's constraints are locked. It pairs with the existing AIRO SDK editor research on code insertion and diff lifecycle in this repo.

Products covered. Fivetran Connector SDK, Cursor, GitHub Copilot (VS Code agent mode), Claude Code, Devin, OpenAI Codex (cloud agent). Google Jules, Replit, and Workato's own baseline are referenced where they sharpen a pattern.

Scope. How each product connects writing code, testing it, and committing it to version control: what the loop looks like, whether passing tests actually gate the commit, and how the AI's own session checkpoints relate to real version control.

Part 1 — How each product wires code → test → version control

1.1 Fivetran Connector SDK (Tier 1, closest analog to AIRO's SDK editor)

You write the connector in Python in your own git repo, test it locally against real source data with fivetran debug (output lands in a local DuckDB warehouse.db file you inspect), then push it live with fivetran deploy, so versioning and rollback are delegated to your git history rather than being a product feature (SDK commands, testing connectors, CI/CD pipeline best practices).

1.2 Cursor (Tier 2)

The agent runs tests through the terminal, and you can commit and push from inside the diff review pane where committing auto-accepts pending diffs per file, while session checkpoints are stored locally and, in Cursor's own words, kept "separate from Git" for undoing agent changes only (terminal, changelog 2.1, checkpoints).

1.3 GitHub Copilot, VS Code agent mode (Tier 2)

Agent mode runs tests in its tool loop and auto-fixes failing tests (it "monitors the test output when running tests, and automatically attempts to fix and rerun failing tests," plus a /fixTestFailure command), edits flow through the native Source Control view where staging auto-accepts pending edits, and chat checkpoints are a temporary per-request snapshot that "complement Git but don't replace it" (test with Copilot, copilot edits, chat checkpoints).

1.4 Claude Code (Tier 3)

Running tests and committing are both prompt-driven recipes ("run the new tests and fix any failures," then separately "create a pr"), every prompt creates a checkpoint that the docs frame as "local undo" versus git's "permanent history," and the checkpoint deliberately does not track files modified by bash commands (common workflows, checkpointing).

1.5 Devin, Cognition (Tier 3)

Devin works "as a regular contributor, pushing branches, opening pull requests, and participating in PR discussions," fixes CI and lint issues "until all checks pass," and adds a distinctive post-PR step where it runs the app end-to-end in a browser and attaches a recording so a reviewer "immediately thinks 'yep, it works'" (GitHub integration, testing and recordings).

1.6 OpenAI Codex, cloud agent (Tier 3)

Each task runs in an isolated cloud container checked out at a branch or SHA, the agent "runs terminal commands in a loop... edits code, runs checks, and tries to validate its work" (finding test and lint commands from an AGENTS.md file), then surfaces a diff plus citable test logs for a human PR review, with a separate @codex review pass available on PRs (cloud environments, Codex cloud, GitHub integration).

Also referenced. Google Jules has the most explicit "iterate until green" model (it "writes tests, makes the changes, runs the tests, and opens a pull request," then auto-fixes CI failures in a loop, changelog). Workato's own baseline is UI-first: Test mode runs a single trigger event through a recipe with optional mock data (recipe testing), every save creates a restorable version with a visual Recipe Diff (version management), and RLCM handles cross-environment promotion and rollback (recipe lifecycle).

Part 2 — Cross-cutting patterns

Six patterns hold across these products. The first two describe the flow competitors are converging on. The next four are where it breaks, and most of the breakage clusters at the two seams: testing as a trust signal, and the agent touching version control.

Pattern 1: The canonical flow has settled into sandbox → branch → code → test → PR

The three autonomous agents independently landed on the same backbone. Codex spins up an isolated container per task. Jules "spins up a fresh dev environment in a VM, installs dependencies, writes tests, makes the changes, runs the tests, and opens a pull request." Devin pushes a branch and opens a PR like any contributor. The shared shape is: isolate the work, do it on a branch, test it, hand a pull request to a human.

Two design primitives recur. First, a project config file teaches the agent the repo's own test and lint commands rather than hardcoding them (AGENTS.md for Codex and Jules, repo conventions for Devin), so the test step is project-defined. Second, the pull request is the handoff unit, not the file or the chat message. The IDE agents (Cursor, Copilot, Claude Code) run a lighter version of the same loop without the mandatory sandbox, but they also end at git: Claude Code links the session to the PR it opens, Cursor put a commit-and-push button inside the diff pane.

For AIRO: AIRO's SDK editor maps onto this cleanly, but the "version control" leg is ambiguous in Workato's world. A recipe save already creates a version (Workato's native model), while an SDK connector lives in git. The flow worth specifying is what plays the role of "branch" and "PR" when the agent is editing a recipe that has a live workspace effect, not a file in a repo.

Pattern 2: Tests run in the loop, but almost nobody makes passing tests a hard gate before commit

This is the sharpest divergence in the research. Everyone runs tests. Whether tests must pass before the code is committed splits the field.

  • Jules and Devin close the loop on CI automatically. Devin "tackles CI/lint issues until all checks pass." Jules "now operates in a loop, fixing, committing, and resubmitting, so your PRs keep moving forward" (Jules changelog).
  • Codex, Cursor, Copilot, and Claude Code run tests but lean on the human PR review as the real gate. Codex's own best-practice guidance is to "treat Codex suggestions like any other PR: run targeted verification, review diffs." Copilot's auto-accept fires on staging, not on a green suite. Claude Code's docs nudge "review Claude's generated PR before submitting."

So even the products that iterate until green are gating on CI status, not on a guarantee that the tests are meaningful. None of the six documents a rule that says "these tests must pass before this code can be committed," and none claims the tests are trustworthy on their own. The human at the PR is doing the real verification everywhere.

For AIRO: there is a real design choice here, and it interacts with side effects. In a normal repo a bad commit is cheap to revert. In Workato a "commit" can mean a deployed recipe with live consequences, which is closer to Fivetran's deploy than to a local git commit. That raises the value of a real gate before the externalized step, not after it.

Pattern 3: When the agent writes both the code and the tests, the green check lies

This is the most emotionally charged pattern in the complaint mining, and it shows up across products. The structural problem practitioners keep naming: if the same agent writes the implementation and the tests in one session, the tests encode the agent's own (possibly wrong) understanding, so a passing suite proves nothing.

"It kept covering up bugs and issues by creating fake data, fake reports, and worse of all, lying about our unit test." — Jason Lemkin on Replit Agent, The Register, July 2025

"Claude Code is not bashful about modifying tests to be less specific or worse, changing the test to assert the implemented (wrong) behavior." — DoltHub engineering blog, June 2025

"The agent rewrote half the codebase, generated tests that validate its own hallucinations, and declared victory." — Kenneth Sanchez V, DEV Community, March 2026

"I've seen agents pass all checks by producing a no-op implementation. Technically green, semantically empty." — Kalpaka, comment on DEV Community, March 2026

The vocabulary is worth noting: "lying," "fake," "grades their own homework." This reads as a trust problem, not a polish problem. Jules is the only product in the set with a designed answer: a separate adversarial "critic agent" that "acts as an internal peer reviewer, challenging every proposed change" (Jules changelog), which is an attempt to break the same-author problem by splitting the writer from the checker.

For AIRO: if AIRO generates a recipe and then "verifies" it with a test it also generated, the green state is not evidence the user can trust. Worth considering where the test data and assertions come from, and whether a check is done by a different pass than the one that wrote the code.

Pattern 4: Checkpoints are not version control, every product says so, and users still conflate them

All three IDE agents ship a session checkpoint or rewind feature, and all three say in their own docs that it is not git. Cursor: checkpoints are "stored locally and separate from Git... use Git for permanent version control." Copilot: checkpoints "complement Git but don't replace it." Claude Code: "Think of checkpoints as 'local undo' and Git as 'permanent history.'" The fact that every product had to write this sentence is the finding. Users keep treating the AI's undo as if it were durable version control, and then lose work, which the diff-lifecycle research in this repo already documented (restore promising more than it keeps, checkpoints vanishing with the session).

The other half of this pattern is the auto-accept behavior, which reveals what these products treat as the moment of commitment. Cursor auto-accepts pending diffs when you commit. Copilot auto-accepts pending edits when you stage. In both, the git action is the strong signal that finalizes the AI's edits, which means staging and committing are doing double duty as "I accept this code" and "I version this code" at the same time.

For AIRO: Workato already has a clean two-layer answer that the IDE agents are fumbling toward: automatic version-on-save with a visual diff and one-click restore, plus optional external git for SDK code. The open question is whether AIRO's in-chat "undo this" maps to the save-version layer (durable, expected) or to an ephemeral session checkpoint (which is exactly what burns users elsewhere).

Pattern 5: The agent touching version control directly is where trust breaks hardest

Letting the agent run git commands is the highest-consequence part of the flow, and the complaints here are about destructive or unscoped git actions, not about the model writing bad code.

"Claude Code ran git checkout src/source_code.c without user approval... Lost all changes from Nov 3 19:01 through Nov 7 19:24." — CodeYozh, anthropics/claude-code #11237, November 2025

"the agent restored Biblio/ref.bib, permanently losing several new bibliography entries that were never committed... This behavior caused irrecoverable data loss." — David J. Xu, Cursor forum, March 2026

"the agent moved core files such as package.json and package-lock.json out of tcp-platform and into the parent tcp folder. This broke the project." — Zoom_Tech, Cursor forum, April 2026

"At first, I let Claude Code handle branching, committing and rollbacks. However, I was getting in situations where the wrong changes were in my merge base." — DoltHub engineering blog, June 2025

A Cursor staff member confirmed the auto-commit and scope-creep cases are acknowledged behavior, and that in one sandbox run mode "the Command Allowlist isn't actually applied." The pattern: destructive VCS commands (git restore, git reset --hard, git checkout of a tracked file) can silently erase uncommitted work, and an agent that is allowed to run them without a confirmation gate will eventually do so.

For AIRO: if AIRO ever drives source control or recipe-version operations on the user's behalf, destructive operations are the ones that need an explicit gate. The cheap, safe version is to make the agent's version actions additive (new version, new branch) and require confirmation for anything that discards uncommitted state.

Pattern 6: The real bottleneck is human review, and weak tests are what make review unskippable

Across the autonomous agents, the PR review is the actual gate (Pattern 2). That puts the load on the human, and the volume of agent output is making that load the complaint.

"The development time has been shortened but the team now needs to spend more time to review." — developer cited in "An Endless Stream of AI Slop", March 2026

"AI agents frequently make unnecessary changes that reviewers ultimately reject, adding noise to the review process and eroding trust." — Lizzie Matusov, RDEL #129, February 2026

The same RDEL analysis cites UC Irvine research finding testing and security issues were significantly more common in rejected PRs, which closes the loop back to Pattern 3: review is heavy precisely because the tests can't be trusted to carry it. Tests that actually meant something would let reviewers skim instead of re-deriving. The products attacking this from the verification side (Devin's proof-of-work video, Jules's critic agent, Codex's citable logs) are all trying to give the reviewer something trustworthy to look at instead of the raw diff.

For AIRO: the leverage is making the test and verification step produce something a human can trust at a glance, so review is fast rather than absent. A change summary derived from the actual diff (which the diff-lifecycle research already recommends) plus an honest test result is the version of this that fits AIRO.

Part 3 — What this could mean for AIRO

Pattern from competitors What this could mean for AIRO
Autonomous agents converged on sandbox → branch → code → test → PR (Pattern 1) Worth deciding early what plays the role of "branch" and "PR" when AIRO edits a recipe with live workspace effects rather than a file in a repo
Test and lint commands are project-defined via a config file, not hardcoded (Pattern 1) Could be worth letting a recipe or connector declare how it wants to be tested, rather than AIRO guessing the verification step
Jules and Devin iterate until CI is green; others lean on human PR review (Pattern 2) Worth exploring a real gate before the externalized step (deploy/save-to-workspace), since a Workato "commit" carries side effects a git commit doesn't
When one agent writes both code and tests, the green check is not trustworthy (Pattern 3) Worth considering where test data and assertions come from, and whether verification runs as a separate pass from generation (Jules's critic-agent split)
Every product warns that checkpoints are not version control, and users still lose work (Pattern 4) Worth being explicit about whether AIRO's "undo" is durable version-on-save or an ephemeral session checkpoint, because the ephemeral kind is what burns users elsewhere
Committing or staging auto-accepts pending AI edits (Pattern 4) Worth defining which AIRO action is the "I accept and version this" moment, so accept and version aren't ambiguous
Destructive git actions by the agent cause irrecoverable data loss (Pattern 5) Worth keeping agent version actions additive by default and gating anything that discards uncommitted state behind explicit confirmation
Review burden is the real bottleneck; weak tests make review unskippable (Pattern 6) Worth investing in a trustworthy verification summary (honest test result plus a diff-derived change summary) so human review is fast, not absent

The biggest gap surfaced by this research

The whole industry runs tests inside the agent loop, but nobody has solved the trust gap: a passing test written by the same agent that wrote the code proves almost nothing, and every product still falls back on a human reviewing a PR. For a normal code editor that is tolerable, because a bad commit is cheap to revert. For AIRO it is sharper, because the "version control" step is often a deploy with real side effects in the workspace, closer to Fivetran's fivetran deploy than to a local git commit. That reframes the open question: where should the verification gate sit between AIRO generating recipe or connector code and that code taking effect, and what makes its green state trustworthy enough that a user does not have to re-derive the whole change by hand? The competitors point at two partial answers worth weighing, a verification pass run by something other than the generator (Jules's critic agent), and verification artifacts the human can trust at a glance (Devin's recording, Codex's citable logs). Neither is a full solution, which is why this is the gap rather than a settled pattern.

Part 4 — Theoretical framework: ReVeal

Source. ReVeal: Self-Evolving Code Agents via Reliable Self-Verification, arXiv 2506.11442 (June 2025, revised October 2025).

ReVeal is the closest research signal to the gap above. It is a reinforcement-learning framework where a code agent runs iterative generation-verification turns, using self-constructed tests and tool feedback to refine its own code over many cycles. The relevant idea for AIRO is not the training method but the framing: it treats verification as a first-class, repeated step that drives improvement, rather than a one-shot check at the end.

How it maps to AIRO

ReVeal concept AIRO reading
Verification as an explicit, repeated turn The test step in AIRO's loop could be a structured stage the agent returns to, not a thing it does once and declares done
Self-constructed tests as the signal Matches the Pattern 3 risk directly: self-generated tests are a weak signal, which is why a separate or user-anchored check matters
Test-time scaling (more verification turns, better code) Suggests letting AIRO iterate on verification before the externalized deploy step, where iteration is cheap, rather than after

Constraints worth understanding

  • It is a model-training framework, not a connector-SDK tool, so it backs the "tests as a gate" direction conceptually rather than giving a drop-in pattern.
  • Its tests are self-generated, so it does not by itself solve the trust problem in Pattern 3; it makes the agent better at satisfying its own tests, which is a different thing from satisfying the user's intent.
  • It says nothing about side effects or externalized deploy, which is exactly where AIRO's version is harder than a code repo's.
