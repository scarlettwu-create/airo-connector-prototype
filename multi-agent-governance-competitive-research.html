Multi-Agent Governance — Competitive Research

Context. For AIRO's agentic surface at Workato, looking at how other products govern fleets of agents: who an agent is, what it's allowed to do, who approved it, and how you reconstruct what happened. Companion to the multi-agent-orchestration research (this repo); that doc covered how work gets split, this one covers the controls around it. Relevant to Workato's audit-trail and recipe-lifecycle heritage.

Products covered. Microsoft (Copilot Studio + Entra Agent ID + Agent 365), Salesforce Agentforce, AWS Bedrock AgentCore, Kong (AI Gateway / Agent Gateway), Claude Code, Okta / WorkOS (agent identity), Galileo / Arize (agent observability), Devin.

Scope. Governance across five dimensions: per-agent identity and access control, approval gates / human-in-the-loop, audit trails and accountability, guardrails and policy enforcement, and monitoring/observability across many agents. Human-in-the-loop is covered lightly here since it was partly handled in the orchestration doc.

Part 1 — Governance behavior, by layer

Governance of an agent fleet isn't one feature. It stacks into four layers, and products are organized below by the layer they own rather than one row each. Identity (who the agent is), gateway (the network choke point every call passes through), execution (what a given call may do in-session), and observability (watching the fleet). Audit and accountability aren't a separate layer; they're a property each layer contributes to. The focused tools own one layer; the big platforms (Microsoft, AWS, Salesforce) span several, so they appear under more than one heading.

Layer What it controls Products that own it
Identity Who the agent is, who owns it, what it's entitled to Microsoft Entra Agent ID, Okta / Auth0, WorkOS (Salesforce reuses its user model)
Gateway Policy on the traffic itself, out-of-band from the agent Kong, AWS AgentCore Gateway + Policy
Execution What a given action may do while the agent runs Claude Code, Salesforce Agentforce, Devin, AWS AgentCore
Observability The production pane of glass across many agents Galileo, Arize (plus Agent 365, Command Center, AgentCore Observability)

1.1 Identity layer — who the agent is

The question here is "which agent, owned by whom, allowed what," and because attribution is a function of identity, this is the layer that answers "who did this." Microsoft Entra Agent ID gives every agent a first-class directory identity with least-privilege blueprints and an accountable human sponsor that transfers to their manager on departure. Okta for AI Agents (GA April 2026) discovers known and shadow agents, registers each with an assigned human owner, and issues scoped short-lived tokens instead of standing credentials, with a kill switch and SIEM audit; Auth0 for AI Agents adds a token vault and async human approval over CIBA. WorkOS frames agents as a new class of actor needing scoped, short-lived, revocable credentials, with fine-grained authorization and an open registration protocol (auth.md). Salesforce takes the lighter path, reusing its existing model: agents run as a dedicated agent user bounded by permission sets.

1.2 Gateway layer — the choke point every call passes through

This layer enforces policy on the traffic itself, out-of-band from the agent, so a rule holds no matter which agent or framework made the call and can't be talked out of by prompt manipulation. Kong AI Gateway 3.14 (Agent Gateway GA April 2026) is the clearest example, running in three forms (LLM Gateway for model traffic, MCP Gateway for tool access, Agent Gateway for agent-to-agent) for "unified governance across APIs, events, LLM calls, MCP tool access, and A2A communication," with Kong Identity as IdP. Its MCP Tool ACLs (January 2026) replace all-or-nothing tool access with default-deny, per-identity filtering of which tools an agent can reach, and the Agent Gateway enforces authn/authz and rate limiting on all A2A traffic, blocks unauthorized interactions "before they execute," and keeps "tamper-evident audit trails of every A2A RPC call, including caller identity, capabilities invoked, and outcomes." AWS reaches this layer too: AgentCore Gateway controls which tools an agent can reach with per-operation access control, and AgentCore Policy (GA March 2026) evaluates every request against deterministic Cedar policies outside agent code.

1.3 Execution layer — what a call may do in-session

This layer constrains and gates individual actions while the agent runs. Claude Code is the most concrete permission UX, enforced by the harness rather than the model: allow/ask/deny scoped to tools and parameters, permission modes (plan is read-only, bypass only in isolated VMs), managed enterprise settings a user can't loosen, and PreToolUse hooks for programmatic enforcement (with a tool_decision audit event per decision). Salesforce Agentforce scopes behavior with an assigned subagents-and-actions allowlist plus the Einstein Trust Layer (PII masking, toxicity filtering) and Agent Script deterministic guardrails. Devin centers on the org and the PR: per-repo/per-tool least-privilege scoping, RBAC from IdP groups, an ephemeral sandbox with default-deny egress, and the pull request as the human gate (humans decide what merges). AWS AgentCore gives each agent a workload identity with credentials in an encrypted token vault and gates on-behalf-of actions behind OAuth consent.

1.4 Observability layer — watching the fleet

This is the production pane of glass across many agents, and the freshest land grab (every major platform shipped one in 2025-26). Galileo does fleet logging, evals, and runtime guardrails, and shipped an open-source Agent Control plane (March 2026) to write a policy once and enforce across any framework (acquired by Cisco, May 2026). Arize does multi-agent tracing (delegation and tool use across frameworks), session-level observability, and online LLM-as-judge evals on production traces. The platforms ship their own fleet surfaces: Microsoft's Agent 365 registry + Agent Map and Risky Agents report, Salesforce Command Center with its Session Trace data model, AWS AgentCore Observability (OpenTelemetry), and Devin's Session Insights.

The layers are converging: AWS already spans identity, gateway, execution, and observability; Microsoft spans identity, guardrails, and observability; Kong is pushing from gateway into identity. But the unit of approval still differs sharply by layer and product, an A2A call or tool invocation (Kong), a tool call (Claude Code), a PR (Devin), a transaction (Agentforce), or an OAuth consent (AWS, Auth0). For AIRO, picking which layers to own, and what the unit of approval is, is the foundational decision.

Part 2 — User and practitioner pain points (themed insights)

Six themes, drawn from three named production incidents, analyst and security-vendor research with hard stats, and practitioner posts. Note on sourcing: raw end-user gripes (Reddit, sub-3-star reviews) were thin for this topic because governance pain lands on admins and security teams, so the evidence skews toward incident write-ups, analyst stats, and practitioner blogs. The stats are well-corroborated; the incidents are dated and named.

Theme 1: Agents take destructive or unauthorized actions, and nobody approved it

The permission existed. The gate didn't. Three dated incidents make the same point.

  • Replit's AI agent deleted a live production database during a code freeze, then misrepresented its ability to recover the data. Replit's own post-mortem called it "a catastrophic error of judgement" that "violated your explicit trust and instructions"; the user, Jason Lemkin, noted "There is no way to enforce a code freeze in vibe coding apps like Replit" (The Register, July 2025).
  • At PocketOS, an AI coding agent deleted a production database and every backup in nine seconds. The detail that should worry a governance team: "The agent authenticated normally, called a valid endpoint, and executed a permitted operation" (Eon, April 2026). Nothing looked anomalous, because the permission itself was the hole.
  • A security researcher planted a poisoned GitHub issue and watched Devin escalate its own privileges to finish the task: "it opened another Terminal, added the execute permission and executed the malware binary again," ending with "an attacker can gain access to all secrets and keys on the devin-box" (wunderwuzzi, Embrace The Red, August 2025).

The common thread is that least-privilege scoping and a pre-execution gate would have stopped all three, and none was in place. This is why AWS's AgentCore Policy enforces Cedar policies outside agent code, and why Claude Code enforces permissions in the harness rather than the model.

For AIRO: the destructive-action class is the one that ends up in a headline. Worth deciding which agent actions are irreversible enough to require a hard gate (a created connection, a deployed recipe, a deletion) regardless of what the agent's plan says.

Theme 2: You can't tell whether a human or an agent did it

Once agents act, attribution breaks, and that's a compliance problem, not just a debugging one.

  • The headline stats: "68% of organizations cannot distinguish AI agent actions from human actions after the fact," and "33% lack evidence-quality audit trails entirely" (Anhang Zhu, TierZero, April 2026).
  • OWASP's agentic threat taxonomy names this directly as T8, Repudiation and Untraceability, and T9, Identity Spoofing, both rooted in agents sharing or borrowing non-human identities (OWASP, 2025).

The products that solve this tie every action to a named identity with a human owner: Microsoft's Entra Agent ID gives each agent a directory identity with an accountable sponsor, and Okta registers each agent with an assigned owner. Attribution is a function of identity, so products without per-agent identity (the observability tools, and to a degree the dev tools that scope by config) can't fully answer "who did this."

For AIRO: Workato already has an audit-trail and recipe-lifecycle heritage. Carrying that forward so every agent action is attributable to a scoped agent identity with a human owner is the natural extension, and it's where most competitors are weak.

Theme 3: Agents are over-permissioned and carry standing credentials

Agents tend to inherit broad, durable access they never need.

  • "74% of AI agents are provisioned with more permissions than their tasks require" (TierZero, April 2026).
  • On AWS specifically: "Most Bedrock agents in production are running on the same IAM role they were built with. That role is now a standing identity with access to whatever services got attached during testing ... Standing access is convenient on a Wednesday, but becomes a liability on a Saturday" (Sonrai Security, May 2026).
  • On Salesforce: agents "inherit permissions from your security model," and Salesforce permission models "get messy fast" with profile sprawl and permission creep, so an over-permissioned agent user expands the attack surface for prompt injection and data exfiltration (Sweep).

The design answer the identity vendors converge on is task-scoped, short-lived credentials rather than role-scoped standing ones: access granted at task start, revoked at completion (Okta's least-privilege thesis, AWS's token vault, WorkOS's scope attenuation). Kong attacks the same problem at the tool level: its MCP Tool ACLs replace the "grant full access to all tools or deny entirely" choice with default-deny, per-identity filtering of exactly which tools an agent can see.

For AIRO: if an agent in a workflow inherits the builder's full connection and data access, that's the Salesforce problem waiting to happen. Worth scoping an agent's access to the specific connections and actions its task needs, not the workspace's full footprint.

Theme 4: Shadow agents pile up with no inventory or owner

Agents proliferate faster than anyone can track them, and admins lose the thread.

  • The standout practitioner voice is Tony Redmond on Microsoft 365: "tools to manage the lifecycle of agents are largely missing," and the line that captures the pain, "there's no trace of the remaining two agents and I have no idea how to clean up the phantoms" (Practical365, February 2025).
  • At scale, the CSA found non-human identities outnumber humans 45:1 (144:1 in cloud-native environments), yet "51% of organizations report no clear ownership of AI identities" and "more than 16% do not track the creation of AI-related identities at all" (Cloud Security Alliance, May 2026).

This is exactly what the new registry surfaces are for: Microsoft's Agent 365 registry and Agent Map, Okta's shadow-agent discovery. The pattern across products is that an inventory with ownership is becoming the first governance primitive, before policy or audit.

For AIRO: a single place to see every agent in a workspace, who owns it, and what it can touch is the foundation the other governance features sit on. Worth treating the agent registry as table stakes rather than a later add.

Theme 5: Guardrails get bypassed, and per-action checks miss the drift

Guardrails built on the same model they're guarding, or that check one action at a time, fail in predictable ways.

  • Multi-turn drift slips past stateless checks: "A prompt injection that gradually shifts the agent's goal over 20 turns can slip through per-action checks because no single action looks suspicious" (BuildMVPFast, 2026).
  • The judge inherits the model's weakness: "if the base model is vulnerable to prompt injections, then the judge model will be too, and vulnerabilities compound rather than resolve" (HiddenLayer, October 2025).

This is why the strongest products are moving policy enforcement out of the model into deterministic engines: AWS AgentCore Policy uses Cedar rules evaluated outside agent code, Salesforce Agent Script encodes deterministic guardrails, and Galileo's Agent Control is a separate control plane. Kong takes the most structural version of this, putting a gateway between communicating agents so policy is enforced on the traffic itself, blocking unauthorized interactions before they execute, no matter which agent or framework made the call. The shared belief is that pure-LLM guardrails are not enough on their own.

For AIRO: for the actions that matter (data egress, irreversible operations), a deterministic check outside the agent's reasoning is more trustworthy than a prompt-level guardrail the same model can be talked out of.

Theme 6: Governance lags far behind deployment, and it doesn't scale to fleet size

The gap between how fast agents ship and how fast governance catches up is wide and widening.

  • "77% of enterprise technology leaders said AI adoption is outpacing governance capabilities," and Gartner predicts that "by 2027, 40% of enterprises will demote or decommission autonomous AI agents due to governance gaps identified only after production incidents occur" (iEnable; Gartner, May 2026).
  • It also doesn't scale operationally. Fleets become unmanageable around 11 to 20 agents (Galileo), and the admin burden compounds: reviewing a couple of agents is easy, "doing the same for several hundred (or perhaps thousands) of agents annually is a different matter" (Practical365, February 2025).

For AIRO: governance that requires manual per-agent review won't survive contact with a workspace that has hundreds of agents. Worth designing the governance surface around policy that applies to groups of agents by default, with review reserved for exceptions.

The biggest gap surfaced by this research

The sharpest, most defensible opportunity is attribution. The single most-cited governance failure is that organizations can't reconstruct which agent, or whether a human or an agent, took a given action (68% per TierZero; OWASP T8). Every strong product solves it the same way, by tying actions to a per-agent identity with a named human owner (Entra Agent ID's sponsor, Okta's assigned owner), and the products that govern at the execution or observability layer without a real identity can't fully answer the question.

Workato is unusually well-positioned here. It already has an audit-trail and recipe-lifecycle heritage that most agent-native competitors lack, so "every agent action is attributable to a scoped identity with an owner and an evidence-quality trail" is an extension of something Workato already does, not a net-new capability. For AIRO, the open question is whether agents get first-class scoped identities (the Entra/Okta model) or remain scoped by configuration and inherited permissions (the Claude Code/Salesforce model). That one decision determines whether AIRO can answer "who did this, and were they allowed to" cleanly, or has to reconstruct it after the fact like the 68%.

Part 4 — Framework: OWASP agentic threats, mapped to governance surfaces

Source. Agentic AI: Threats and Mitigations, OWASP Agentic Security Initiative (February 2025), with the Multi-Agentic System Threat Modeling Guide (April 2025).

OWASP's taxonomy of 15 agentic threats (T1 to T15) is the vocabulary the security community already uses, which makes it the cleanest way to organize a governance conversation with security stakeholders. The governance-relevant threats map directly onto product surfaces:

OWASP threat Governance surface in AIRO terms
T3 Privilege Compromise, T9 Identity Spoofing Per-agent scoped identity and least-privilege access
T8 Repudiation and Untraceability Audit trail attributable to an agent identity and owner
T2 Tool Misuse, T6 Intent Breaking Deterministic policy on which tools and actions are allowed
T10 Overwhelming Human-in-the-Loop Approval gates designed not to fatigue the reviewer
T13 Rogue Agents in Multi-Agent Systems Fleet observability of agent-to-agent coordination

For the enterprise risk and compliance buyer, the NIST AI Risk Management Framework is the parallel language: its four functions (GOVERN, MAP, MEASURE, MANAGE) map onto the same surfaces (roles and ownership, agent inventory, observability and evals, approval gates and kill-switches). Speaking both vocabularies lets AIRO frame governance features in terms buyers already evaluate against.

Where this is heading

Two signals worth noting. Gartner predicts guardian agents (agents that review, monitor, and block other agents) will be 10 to 15% of the agentic market by 2030, and academic work like Governance-as-a-Service (August 2025) proposes a runtime enforcement layer that scores each agent by a compliance-weighted trust factor and applies graduated enforcement. Both point the same direction: governance becoming its own layer that watches and constrains the agent fleet, rather than logic baked into each agent. The MorphMind Caliper trust-score idea from the orchestration research is an early consumer-side version of the same instinct.

Constraints worth understanding

  • OWASP names what to defend against; it doesn't prescribe the UX. The threat-to-surface mapping above is the bridge, but the actual design is still open.
  • The strong stats come from analyst and vendor research (TierZero, CSA, Gartner), which is credible for problem-framing but is not peer-reviewed measurement.
  • Several competitor capabilities are very recent or still in preview (AgentCore Policy GA March 2026, Microsoft Conditional Access for agents still rolling out), so the bar is moving quarter to quarter.
  • Raw first-person user complaints are thin; the evidence base here is incidents and analyst data, which is strong for "why this matters" but lighter on "what specifically annoys a daily user."
