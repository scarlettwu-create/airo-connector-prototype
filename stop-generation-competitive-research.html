Stop Generation UX — Competitive Research

Context: Research to inform the AIRO Stop Generation feature (PRD: AIRO UX improvements — Batch 1, Section 1.3). First compiled May 2026, refreshed June 10, 2026. The feature has since moved to its own ticket, AAI-1501 under PMO-2975, targeting June 2026.

Products covered: Relay.app, Zapier Agents, n8n, Replit Agent, Lovable, v0 (Vercel). Figma Make was covered in the May version; its key findings are retained in Part 2 with a June update.

Scope: How each product handles user-initiated stop mid-generation, covering what happens to the artifact, the state, the credits, and the user trust.

What changed in this refresh:

  • Added three Tier 1 workflow automation competitors (Zapier Agents, n8n, alongside Relay.app). The May version skewed toward AI coding agents. The Tier 1 finding is significant: stop is underbuilt across AIRO's own category (new Theme 5).
  • Corrected v0: the stop button reliability bug was patched by Vercel on March 20, 2026. The May version recorded it as unresolved.
  • Lovable shipped explicit stop semantics on May 21, 2026 ("file edits now persist when stopping the agent").
  • Figma Make added queued messages and plan mode on June 3, 2026, a steer-instead-of-stop pattern. Still no stop button.
  • Added the InterruptBench paper (April 2026), the first benchmark for interruptible agents.

Part 1 — Stop Generation Behavior (Official Documentation)

1.1 Relay.app

Three-tier stop: Pause (resumable), End (terminal), and Freeze all runs (workspace-wide kill switch), with no rollback for external side effects already executed. No changes to stop behavior in the May-June 2026 changelog.

1.2 Zapier Agents

Cancelled is one of six documented task statuses ("You cancelled the task"), but the cancel affordance itself is undocumented for triggered runs; the only documented stop control is a Stop button in the chat field for AI generation. Runaway protection is a budget guardrail rather than a kill switch: a per-run activity cap (10 on Free, 40 on paid) pauses the agent and requests user input before continuing, and overrun activities are not charged. Nothing is documented about partial-work handling or undo after cancel.

1.3 n8n

Stop exists only in the UI, via a stop button in the All Executions view (which the docs page itself doesn't document) plus a canvas stop during manual testing; the button has been missing from the workflow-specific view since users first asked in March 2024. There is still no public API cancel endpoint as of June 2026; a community PR adding one has been open since October 2025. Side effects from already-executed nodes simply persist after stop, with no checkpointing or rollback documented.

1.4 Replit Agent

Stop button at the top of the chat halts the agent immediately with auto-created checkpoints for one-click rollback, though closing the tab is a known bug that leaves the agent running in the background. No stop-related changes in May-June 2026 updates.

1.5 Lovable

Stop button inline in the chat halts immediately, with a one-click "Undo latest edit" button appearing right next to the stopped response card. As of May 21, 2026, Lovable made its stop semantics explicit: "File edits now persist when stopping the agent," so stop keeps partial work rather than discarding it.

1.6 v0 (Vercel)

Stop button was historically unreliable, sometimes triggering full app rewrites instead of halting, until Vercel patched it on March 20, 2026 after sustained community pressure; chat-preview desync after stop remains a reported issue. On June 8, 2026, v0 moved the other direction on involuntary stops: long agent runs now continue automatically instead of stopping at the timeout.

Note on Figma Make (covered in the May version): still no stop button, but on June 3, 2026 it added queued messages (stack follow-up instructions while generation runs) and an opt-in plan mode (edit and approve a plan before generation starts). Steering and upstream prevention instead of mid-run stopping.

Part 2 — User Pain Points (Themed Insights)

Five themes emerged from mining complaints across Reddit, official forums, GitHub, G2, Product Hunt, Hacker News, and community Discords. All source links are in the appendix. The themes lead directly into the AIRO design implications in Part 3.

Theme 1: Stop button doesn't actually stop

Users click stop and either nothing happens, the stop itself errors, or the agent does more work after the click. The refresh adds Tier 1 evidence: in workflow automation products, stop doesn't just lag, it sometimes fails outright.

  • v0 (March 2026): "When v0 starts going on a tangent and I click STOP, it doesn't stop and it just completely re-writes your entire application... It stops after it puts the 'Fixes' that I nor anyone else is asking it to fix." (thread) Vercel acknowledged and patched this on March 20, 2026, with users confirming the fix the same day. Worth noting how long it shipped broken first.
  • v0 API: CTRL+C or Postman cancel do not stop server-side generation; the platform processes asynchronously and ignores client disconnect (thread).
  • Replit: User wrote "DO NOT CHANGE FROM RADAR, THIS IS NOT A FIX" in all caps in the prompt. Agent ignored and swapped the implementation across multiple checkpoints anyway (thread).
  • Replit: "After using agent, I can close the agent tab and the icon on left hand side still shimmers... it has not stopped." (thread)
  • n8n: "There seems to be a bug where an execution is stuck in Running state and cannot be stopped. When I try to stop it, I get prompted with a 'Problem stopping execution'." (thread, September 2024)
  • n8n: "I accidentally triggered a workflow that created around 1000 asynchronous executions. Now my instance is stuck with endless running executions, and I can't stop them all manually from the UI." (GitHub issue, September 2025, closed "not planned")
  • Zapier Agents: "It's been stuck at this step for 30 minutes. Anyone else experiencing this issue with agents in Zaps?" (thread, May 2025)

For AIRO: the PRD's under-2-seconds halt requirement is the right bar, and the v0 arc shows the cost of shipping below it. A stop that errors ("Problem stopping execution") is worse than a slow stop; the failure path of the stop action itself deserves a defined UX.

Theme 2: Credits and money burned on stopped or broken runs

The #1 grievance across credit-based products. Users expect "stop equals no charge for unfinished work." Reality is the opposite, and the financial damage is real.

  • v0: One user reported losing $60-90 in two days, then $210 in another two days. Called the experience "extractive" and said "It happens for 30-40% of my prompting." (thread)
  • Replit: The $607 bill case. Developer launched Agent, went to lunch, agent ran 2 hours in a retry loop. "No hard stop. No popup. No automatic pause at $100, $200, or $500." (blog post)
  • Replit: Another user reported "charged $28 for 7 MINUTES of work" and "spent over 2k last month" (thread). Per Replit's own billing docs: "All Agent interactions are billable... in cases where the Agent performed work but didn't make code changes, there is still a charge."
  • Lovable: "I've spent over 20 credits trying to fix a code that results in my app being frozen on a 'loading page'." Another user: "I still had 360 credits that I had already paid for, and after I canceled my subscription, I had only 5." (source) Per shipper.now, each "Fix" attempt counts as a new paid generation even when the bug was caused by Lovable itself.
  • Figma Make: Reports of 700+ credit prompts that produce nothing, 62 credits to change three numbers, and entire credit balances wiped out without a single successful change (thread). Still ongoing as of May 2026: "I want to request a review of AI credits used during repeated troubleshooting on my project." (thread)

Vercel eventually promised forward-looking "you'll not be charged for a failed execution in the future," signaling that no-charge-on-failure is becoming table stakes for credit-based products. Zapier Agents goes further with a structural guardrail: a per-run activity cap that pauses the agent and asks for input before continuing, with overrun activities not charged. Prevention by budget rather than reaction by kill switch.

Theme 3: Stop creates orphan state, and products hide it

This is the most damaging pattern across products. Stop creates partial state in the workspace, the product doesn't tell the user about it, and trust collapses when users discover the orphans later. The emotional vocabulary in user complaints is the strongest signal that this is a trust problem, not a polish problem.

Concrete examples of broken or hidden state after stop:

  • v0: Chat panel code and project preview code frequently end up out of sync after stop. No "Stopped" indicator, so users have to infer the stop happened.
  • Figma Make: "Version Success" in chat alongside "Build Error" in preview is a common failure pattern (thread).
  • Replit: Checkpoints get created even when no code changes happened, confusing users about what was actually done.
  • Figma Make: Users give up entirely on projects after repeated state-corruption incidents.
  • n8n (April 2026): "Some records get duplicated. Some records are skipped. I can't reliably continue from where it stopped... There's no real checkpointing, so if it crashes at page 20, I have to restart from page 1." (thread) The same partial-state problem AIRO's orphan recipes represent, in AIRO's own product category.

One product has started addressing this by making the contract explicit rather than fixing the state: Lovable's May 21, 2026 changelog announces "File edits now persist when stopping the agent," with no extra action needed. Users know exactly what stop does to their work. None of the other products researched state their stop-state contract anywhere.

The vocabulary users reach for tells you it's a trust issue, not a slowness issue:

  • "Extractive" (v0)
  • "I give up" (Figma Make, multiple users)
  • "Hero to zero" (Lovable)
  • "Feel cheated" (multiple)
  • "I had to move my project out" (Figma Make)

The takeaway: expose the mess, don't hide it. Users will forgive a stopped task. They won't forgive finding orphan artifacts later that nobody mentioned. List what got built, name what didn't, recommend cleanup if needed. This is the single most important design pattern to take from this research, and it shows up again in the Part 3 implications.

Theme 4: What users want is "Stop AND..." verbs, not just stop

Across all the complaints, users don't actually want a Stop button alone. They want stop combined with another verb. Four "Stop AND..." actions surface across the research, and each has a direct implication for AIRO.

Stop AND don't charge me

Users universally expect "stop equals no charge for unfinished work." Reality across credit-based products is the opposite: Replit charges for "Agent interactions" even when no code changes happened, Lovable charges every "Fix" attempt as a fresh generation, v0 burns credits on stopped or broken runs. Vercel eventually promised forward-looking "you'll not be charged for a failed execution in the future" — signaling no-charge-on-failure is becoming table stakes for AI products.

For AIRO: if AIRO ever moves toward credit-based billing, the stop-billing contract is worth defining upfront. Even without credits, the principle applies: stop should not feel punitive.

Stop AND tell me what got built

No competitor sends a structured follow-up listing what was completed vs. left incomplete. Users have to discover the state of the workspace themselves, which is exactly how the trust-collapse pattern from Theme 3 starts.

For AIRO: the PRD's June 4 update replaces the post-stop summary message with an in-flight plan artifact showing completed (✓) and pending (○) assets in real time, which on stop freezes at the final state. That's ahead of every product researched. The open question remains what happens to the partially built recipes: the PRD currently says they are "not surfaced," which is the hide-the-mess pattern Theme 3 warns about. Worth considering whether the stopped plan artifact should name the orphans and recommend cleanup, even if one-click cleanup isn't feasible yet.

Stop AND undo cleanly

Only Lovable has a one-click "Undo latest edit" button on the stopped artifact card. Replit has rollback to checkpoints but it's manual and at a different granularity. Most other products have no undo at all, so partial state lingers.

For AIRO: Lovable's pattern is the cleanest reference for AIRO's stopped Build All card. One click to revert the partial state would directly address the orphan empty recipe pain in the current PRD.

Stop AND let me redirect

At the time of the May version, none of the products researched supported "transition the agent to a new prompt without abandoning the chat." That's starting to move. Figma Make's June 3, 2026 release added queued messages, letting users stack follow-up instructions while generation runs (steer, but not interrupt). v0 went the other way on June 8: long runs now auto-continue past the timeout, reducing involuntary stops but adding none of the voluntary kind. And the InterruptBench paper (April 2026) formalizes exactly this: it distinguishes three mid-task interruption types (addition, revision, retraction) and finds that handling them well is still hard even for large models.

For AIRO: the PRD already scopes send-new-message-as-stop, which goes further than anything shipped by the products researched. InterruptBench's three interruption types are a useful design check: a new message can add to the task, revise it, or retract it entirely, and the right stop behavior (keep partials, discard, or redirect context) differs per type.

Theme 5: In AIRO's own category, stop barely exists

This is the new Tier 1 finding from the refresh. The workflow automation products closest to AIRO treat stop as ops tooling rather than a designed user experience, and users notice.

  • n8n (May 2025): "I'm honestly flabbergasted that there still exists NO way to cancel a running execution via the n8n API." (thread) The community PR adding a public stop endpoint has been open since October 2025.
  • n8n (March 2024): the stop button exists in one view but not another. "Can we get feature parity here?" A second user: "Actually, it's more like a bug in UI, that there is no stop button for waiting executions." (thread) Still open two years later.
  • Zapier Agents: a Cancelled status exists in the docs, but how to cancel a triggered run mid-execution, what happens to partial work, and whether consumed activities are refunded are all undocumented.
  • Relay.app is the exception with its explicit Pause / End / Freeze model, though it leaves external side effects unresolved.

The contrast with Tier 2 is stark: coding agents (Replit, Lovable, v0) have all iterated publicly on stop because their users sit and watch generation happen. Automation products assume runs are unattended, so stop never got designed. AIRO sits exactly at the intersection: a conversational agent whose output is automation infrastructure.

For AIRO: a stop experience that meets the PRD's current bar (always-visible button, under-2s halt, visual stop signal, send-as-stop) would be ahead of every Tier 1 competitor researched, not just at parity. This reframes stop from catch-up work to a differentiation opportunity worth naming in the AAI-1501 context.

The biggest gap surfaced by this research

AIRO doesn't yet have a path to roll back partial artifacts after stop. The "Build All → orphan empty recipes" trade-off in Section 1.3 is the same shape of state corruption that drives Lovable and Figma Make users to abandon projects, and the refresh found it live in AIRO's own category: n8n users restarting 20-page jobs from page 1 because "there's no real checkpointing."

The Atomix paper (§4) offers one possible direction here: treat the 4 output types as bufferable vs. externalized, defer commit until the transaction is safe, and run compensation handlers on abort. The skepticism this direction gets in practice is also worth knowing: when a transactional agent sandbox launched on Hacker News in May 2026, the top criticism was that versioned state can't undo external side effects, which is precisely the externalized-effects half Atomix addresses with compensation. The current "accepted trade-off" framing in the PRD might be worth revisiting as an "open architectural question with a known direction of investigation," especially given that no shipping product has solved this and there's potential differentiation value in being the first.

Part 4 — Theoretical Framework: Atomix Transactional Effects

Source: Atomix: Transactional Effects for Agent Tool Calls, arXiv 2602.14849 (Feb 2026). HTML version.

This paper explores a question relevant to AIRO: how could agent tool calls behave more like database transactions, so stop produces a clean state instead of orphan side effects? It's worth a read because it's the only piece of work I've found that takes this problem seriously, with benchmarks and a proposed runtime.

How the taxonomy could map to AIRO's output types

The paper splits all tool-call effects into two categories. Mapping them onto AIRO's output types gives an interesting framing:

AIRO output type Atomix category Implication for stop
Blueprint (draft architecture) Bufferable Discard the buffer. Zero side effects. Aligns with current PRD.
Recipe edit Bufferable Revert to last saved. Aligns with current PRD.
New recipe from blueprint Externalized A compensation handler could clean up the orphan recipe on stop.
Build all Externalized (multiple parallel transactions) Completed transactions commit. Aborted ones could be compensated.

Bufferable means changes that can be delayed and held invisible until commit (local file writes, in-memory state). Externalized means changes that execute immediately and would need explicit tracking to undo (API calls, DB writes, emails). The orphan empty recipe problem sits in the Externalized bucket.

Reversibility layer on top: - Reversible: file edits, DB writes, recipe creation. Can be compensated cleanly. - Reversible-with-cost: API calls that consume rate limits or connection slots. - Irreversible: emails sent, real webhooks fired, transactions completed. These would call for explicit gating (user confirmation before execution) rather than after-the-fact rollback.

How compensation could address the orphan recipe trade-off

The current PRD treats orphan empty recipes as an accepted trade-off. The paper proposes a different angle: every tool call could record (scope, idempotency key, compensation handler) when it executes. On stop, the runtime would invoke compensations in reverse dependency order. Failed compensations retry; unresolvable ones could surface to the user with a manual cleanup prompt.

For AIRO this would mean each recipe-building API (create_recipe, link_to_asset_map, add_step, configure_connection) could be paired with a corresponding undo API (delete_recipe, unlink, remove_step, revoke_connection).

The paper's benchmark is striking. In a test of 1,200 irreversible operations, the transactional version leaked 0 effects on abort while the baseline (no transactions) leaked all 1,200. Translated loosely to AIRO: if Build All tried 1,200 recipes and the user stopped at 60%, the current architecture would leave 720 orphans behind. With compensation, that number could be much closer to zero. Runtime overhead is reported at <0.01% of tool latency, so the main investment is in writing the undo APIs rather than runtime performance.

Constraints worth understanding

Three things worth keeping in mind if this direction gets explored further: 1. Atomix is currently a single-process prototype. AIRO runs across AI, Platform, and Conversation Hub, so adopting this pattern would involve solving the distributed frontier tracking problem that the paper itself calls out as future work. 2. Each tool would need a paired undo API. For AIRO, that's roughly 20 to 30 paired endpoints depending on scope. 3. Truly irreversible operations like real webhooks and real emails are different. They benefit from user confirmation before execution rather than after-the-fact rollback. If Build All ever actually runs a recipe (rather than just creating one), that operation falls into this bucket.

New since May: InterruptBench

A second relevant paper appeared in April 2026. InterruptBench ("When Users Change Their Mind: Evaluating Interruptible Agents in Long-Horizon Web Navigation") is the first benchmark for agents that get interrupted mid-task. Its useful contribution for AIRO is a taxonomy of three interruption types: addition (new requirement on top of the task), revision (change something in flight), and retraction (abandon part or all of it). The paper finds even large models handle these poorly. For AIRO's send-new-message-as-stop behavior, the taxonomy is a design check: the right handling of partial work differs depending on which of the three the user's new message is.
