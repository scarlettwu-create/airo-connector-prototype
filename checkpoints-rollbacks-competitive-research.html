Checkpoints and Rollbacks — Competitive Research

Context. Early-stage research for AIRO (Workato's agentic builder), to inform how an AIRO agent should let users roll back after it has made a series of edits in a session. No PRD yet; this is the research that comes before constraints are set.

Products covered. Cursor, Claude Code, GitHub Copilot, Replit Agent, Lovable, n8n. Workato's recipe versioning and Windsurf/Cascade come in as baseline and contrast.

Scope. Agent-session checkpoints and rollback: the agent makes a run of edits (files, recipes, connections, deployments), and the user wants to return to an earlier point. How each product creates checkpoints, what a "restore" actually reverts, what it misses, how long checkpoints live, and where users get burned.

Part 1 — Checkpoint and rollback behavior per product

1.1 Cursor

Auto-creates a checkpoint before significant edits and lets you click "Restore Checkpoint" to revert all modified files, but checkpoints are stored locally, explicitly separate from git, and meant only for undoing agent file edits (not terminal or git operations).

1.2 Claude Code

Every prompt creates a checkpoint and /rewind offers "restore code," "restore conversation," or both, persisting across sessions and auto-cleaned after 30 days, but files changed by bash commands (rm, mv, cp) are not tracked and cannot be undone.

1.3 GitHub Copilot (VS Code)

Opt-in chat checkpoints snapshot affected files before each request; restoring reverts the workspace, removes that request from chat history, and a Redo recovers the undone changes, but checkpoints are scoped to a single chat session and described as temporary.

1.4 Replit Agent

Automatically checkpoints a full app snapshot (files, packages, config, conversation, agent memory, and database contents) at milestones and rolls back instantly forward or backward, though the database is excluded from a rollback unless you opt in under "Additional rollback options."

1.5 Lovable

Keeps a built-in version history of every AI edit where a restore creates a new edit card rather than destroying history (git-revert style), and since Feb 23, 2026 stopping mid-task keeps all completed work.

1.6 n8n

Has no native checkpoint or rollback; the closest is a "Save execution progress" setting that resumes a workflow from where it errored, with true rollback left to user-built compensating sub-workflows.

Baseline and contrast. Workato today versions every recipe on save with a Recipe Diff and "Restore this version" action. Windsurf/Cascade reverts a step's file changes but its own docs warn "Reverts are currently irreversible, so be careful!"

Part 2 — User pain points (themed insights)

I mined the Cursor forum, claude-code and microsoft/vscode GitHub issues, Replit Discourse, the n8n community, and the Vercel community, plus the existing complaint library. Five themes carry the most weight, ordered roughly by how often they show up and how much trust they cost.

Theme 1: Rollback rolls back the code, not the world

Restoring a checkpoint reliably reverts the files the agent edited. Anything with a side effect outside those files (a database, a deployment, a created resource, a shell command) stays changed, and users find out after the fact.

  • Replit: rollbacks exclude the database by default; you have to opt in to include it.
  • Replit: "That rolls back the deployment not the code" — kody-replit, Replit Discourse, April 2025.
  • Replit: "I asked agent to add something to the database and it erased the existing database (said its unrecoverable) and it started an entirely new database. This was detrimental to weeks of work." — andysamland, Replit Discourse, February 2025.
  • Lovable/Supabase: "But I am wondering about what happens to my supabase migrations? Ideally those should be reverted complying to 'Restore' action." — Mohammed Zeeshan, Vercel Community, June 2025.
  • Claude Code: files modified by bash commands are not tracked; "When the response rolls back, those side effects persist but the user has no record of what was done." — martinogdenlabs, claude-code #29684, February 2026.

An AIRO agent that creates a recipe, a connection, or a deployed job has produced exactly this kind of side effect. Rolling back the chat won't un-create the recipe sitting in the workspace.

For AIRO: This is the sharpest decision in the space. Worth settling early what a rollback covers when the agent has already made real workspace changes, and saying it plainly in the restore UI rather than letting users learn the boundary by getting burned. See Part 4 for the bufferable-vs-externalized framing.

Theme 2: Nobody can predict what a "restore" actually reverts

"Restore" means different things across products (conversation only, code only, both, deployment, agent memory, todo state), and the affordances diverge even inside one product. Users can't tell what state they'll land in.

  • Cursor: "It's super confusing that rerunning prompt does different thing than clicking 'restore' checkpoint below that prompt, and that clicking on text below prompt does other changes." — Yuri, Cursor forum, March 2025.
  • Claude Code: "After using /rewind to restore the conversation to a previous checkpoint, the TODO list is not reverted... Claude may repeat tasks it had already completed before the rewind." — FelixRuiGao, claude-code #41542, March 2026.
  • GitHub Copilot: "the memory files (agent-generated context/logic files) created during the discarded portion of the conversation are not deleted or reverted." — bnn1, vscode #307617, April 2026.
  • Cursor: restore "appears to restore the checkpoint as of when the question was completed, not when the question was started." — Charles, Cursor forum, March 2025.

The pattern is partial restore. The agent's state is made of several layers (chat, code, todo, memory, created assets) and a restore touches some but not all of them, which leaves the session in a state the user didn't ask for and can't reason about.

For AIRO: If an AIRO checkpoint spans chat plus recipe edits plus created assets, the restore dialog could name exactly which layers come back. The failure here isn't missing rollback, it's rollback whose scope the user has to discover the hard way.

Theme 3: All-or-nothing, so good work gets thrown out with the bad

Checkpoints attach to whole prompts, so the only way to drop a bad change introduced late is to discard the good progress that came with it. Restore granularity rarely matches how work actually accumulates.

  • Cursor: "You can no longer restore checkpoints at certain points during the process, only with each prompt. That is… bad." — SThompson, Cursor forum, March 2025.
  • Cursor: "there are situations where we might have partially successful code updates, but then extra unwanted elements get introduced in the later stages." — m-akinari, Cursor forum, March 2025.
  • n8n: "There's no real checkpointing, so if it crashes at page 20, I have to restart from page 1." — Keira_Becky, n8n community, April 2026.
  • Make.com (adjacent, replay): "If 8 cases failed, I cannot play the 5th one only it seems. Is there any way to decide which one to execute?" — Gaurav_Srivastava, Make.com community, May 2025.

For AIRO: Worth exploring whether a rollback can target a single recipe or a single step the agent touched, rather than only rewinding the whole session to a prompt boundary.

Theme 4: The safety net isn't safe

The feature users lean on as insurance is itself unreliable: clicks that do nothing, checkpoints that vanish on session end or app update, and reverts with no way back.

  • Cursor: "when I click Rollback (or try to restore a checkpoint), nothing happens and my code stays in the modified" — Bogdan_Moise, Cursor forum, July 2025.
  • Cursor: after a version update "we were unable to click 'Restore Checkpoint', and several files were deleted... it's now impossible to find the files that were deleted after the update." — Kiziev, Cursor forum, July 2025.
  • Replit: "I started a new chat and lost all of the rollback options." — mkennventure, Replit Discourse, January 2025.
  • GitHub Copilot: restoring at a previous point works, but switching to another session and back makes the rolled-back discussion reappear instead of staying at the checkpoint. — garretwilson, vscode #319225, May 2026.
  • Cursor: the "Restore checkpoint" prompt said "you can always undo this later," and "It seems you cannot undo this later." — Ursalbus, Cursor forum, August 2025.

The cleanest counter-example is Copilot's Redo, which reapplies undone changes after a restore. Windsurf/Cascade sits at the other end with reverts its own docs call irreversible.

For AIRO: A rollback users can't trust is arguably worse than none, because they stop keeping their own backups and then lose work. Two things seem worth pinning down early: whether the rollback itself is reversible (a redo), and how long a checkpoint lives (does it survive closing the session).

Theme 5: In metered products, the checkpoint is itself a charge

In credit-based agents, creating a checkpoint can cost money, so an agent that checkpoints aggressively, or checkpoints work that changed nothing, runs up the bill on its own.

  • Replit: "It's now doing multiple checkpoints without changing any code. Something is wonky." — realfunnyeric, Replit Discourse, May 2025.
  • Replit: "The agents tendency to go 'wild' with multiple check points... quickly becomes a vicious cycle that increases cost" — slin175, Replit Discourse, May 2025.
  • Replit: "the agent decided on action but did not implement anything. It created a checkpoint and charged my account." — joeschmoeno, Replit Discourse, October 2025.
  • Replit: "A lot of checkpoints are getting billed, but is all useless because of this" (once the session ended). — NihalAlgo, Replit Discourse, February 2025.

Honest caveat: all four quotes are Replit, because Replit ties checkpoints directly to billing units. It's a single-product pattern today, but it's structural to any metered agent.

For AIRO: If AIRO checkpoints ever get metered, worth deciding whether a checkpoint that produced no change is billable, and whether users can see what a checkpoint costs before it accrues.

Part 3 — AIRO design implications

Competitor pain point What this could mean for AIRO
Rollback reverts files but leaves databases, deployments, and created resources changed (Replit, Lovable, Claude Code; Theme 1) Worth defining what a rollback covers when the agent has already created a recipe, connection, or job, and surfacing that boundary in the restore UI
"Restore" silently means different things (conversation vs code vs memory vs todo) (Cursor, Claude Code, Copilot; Theme 2) Could be worth naming the exact layers a restore reverts in the dialog itself, so scope isn't learned by surprise
Checkpoints only attach to whole prompts, so dropping a late mistake discards good work (Cursor, n8n; Theme 3) Worth exploring per-recipe or per-step rollback rather than only session-level rewind
Restore clicks do nothing, or break after an app update (Cursor; Theme 4) Reliability of the restore action seems worth treating as a core requirement, not a nice-to-have
Checkpoints disappear when the session ends or a new chat starts (Replit, Copilot; Theme 4) Could be worth making checkpoint lifetime an explicit, stated decision rather than an implementation side effect
Reverts with no redo permanently destroy forward history (Cursor, Windsurf; Theme 4) Worth considering whether a rollback is itself reversible, following Copilot's Redo pattern
Restoring mid-run doesn't stop the agent, which keeps editing on top of the reverted state (Copilot) Worth deciding what a restore does to an in-flight run (halt first, then revert)
Aggressive or no-op checkpoints run up the bill (Replit; Theme 5) If metered, worth defining whether no-change checkpoints are billable and showing checkpoint cost upfront
n8n and Tier 1 peers have no real session checkpoint at all (Theme 3) A conversational builder that ships reliable, scoped rollback would lead the workflow-automation category rather than catch up to it

The biggest gap surfaced by this research

Every product here does a decent job rolling back files. None has a clean answer for rolling back the things an agent creates that live outside those files: a deployed app, a database migration, a remote resource. Replit excludes the database from rollback by default, Lovable users ask what happens to their Supabase migrations, Claude Code can't undo a rm, and the practitioner skepticism on the transactional-sandbox thread lands on the same point every time.

For AIRO this is the whole ballgame, because the agent's main output is exactly these externalized effects. A recipe created in the workspace, a connection authorized, a job deployed. Rolling back the conversation does not un-create them. The open question worth answering before the PRD hardens: when the user clicks "restore," what happens to the real resources the agent already created, and does the UI promise rollback or honestly offer compensation (a follow-up action that undoes the effect)? There is also no academic study yet of how users mentally model an agent-session checkpoint versus git or explicit versions, so the mental-model burden falls entirely on the product.

Part 4 — Theoretical framework: Atomix

Source: Atomix: Transactional Tool Calls for Reliable Agentic Workflows, arXiv 2602.14849 (February 2026). HTML version.

Atomix borrows from database transactions and Saga-style compensation to ask when it is safe for an agent to commit an action. Its useful move for AIRO is splitting effects into two kinds: bufferable effects that can be held and discarded cheaply, and externalized effects that hit the outside world immediately and can only be undone by a compensating action.

How the framework maps to AIRO

AIRO output Atomix category Rollback reality
Blueprint / draft plan Bufferable Safe to discard; true rollback works
Unsaved recipe edit Bufferable Discard before save; true rollback works
New recipe or connection created Externalized Already lives in the workspace; needs compensation, not rollback
Deployed job / "build all" Externalized Maximum side effects; compensation is the only honest undo

The line between rows three/four and rows one/two is the line between "we can roll this back" and "we can only offer to undo it with a new action." Showing users which side of that line each output sits on, before they commit, is the design payoff.

Constraints worth understanding

  • Compensation is not free or always possible. Some externalized effects (a sent notification, a consumed credential) have no clean inverse.
  • Resume-after-rollback carries its own risk. ACRFence (March 2026) shows agents regenerate slightly different requests after a restore, which can create duplicate resources. Replaying exact prior parameters is safer than re-generating.
  • Durable resume needs external state. Microsoft's orchestration guidance notes that pause-and-resume requires checkpointing execution state to durable storage, not in-memory context.
  • The framework says nothing about the user-facing mental model, which is the part most likely to bite. See the biggest-gap note above.
