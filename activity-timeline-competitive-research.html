Activity Timeline — Competitive Research

Context. Research compiled to inform a future AIRO PRD on real-time agent activity UI (what users see while AIRO is working). No PRD exists yet; this research is at the discovery stage. Updated June 2026 to add the workflow-automation tier (n8n, Zapier, Make), which is AIRO's direct competitor category.

Products covered. Cursor, Claude Code, Replit Agent, Devin, Lovable (AI coding agents). n8n, Zapier, Make (workflow automation, Tier 1 for AIRO).

Scope. Two related surfaces. First, what users see while the agent is working: thinking stream, tool calls, planned steps, sub-tasks, progress indicators, interaction model. Second, how they review what happened after a run: run history, execution detail, rerun. The coding agents center on the live stream; the workflow-automation tools center on run history, which is the surface AIRO will lean on most.

Part 1 — Activity timeline behavior per product

1.1 Cursor

Cursor's Agent (the multi-step mode inside Composer, distinct from plain Chat) shows live activity inline in the Composer pane: reasoning blocks, file reads, file edits with diffs, and terminal commands stream as they happen. Cursor 2.0 added a dedicated Agents sidebar that lists every agent currently running locally or in the cloud, with each agent's worktree, plan, and tool calls accessible from one place. A Plan Mode (Shift+Tab) generates a markdown todo list before code is written, and a "compact / balanced / detailed" tool call density setting lets users dial down the noise.

  • Agent Mode + Composer overview: cursor.com/product
  • Cursor 2.0 changelog (sidebar, plan mode, multi-agent): cursor.com/changelog/2-0
  • Plan Mode docs: docs.cursor.com/agent/planning
  • Agent best practices: cursor.com/blog/agent-best-practices

1.2 Claude Code

Claude Code streams every tool call inline in the conversation with color-coded badges (Read = blue, Write = green, Edit = amber, Bash = red), each expandable to show full input and output. Extended thinking blocks appear inline as collapsed cards that can be opened individually or toggled with Ctrl+O. The CLI streams bash output token-by-token; the VS Code extension buffers it until the command finishes, which has been a recurring complaint. There is no dedicated planning panel beyond the conversation itself, though Claude often emits a todo list as a tool call when tasks get complex.

  • Tool system overview: callsphere.ai/blog/claude-code-tool-system-explained
  • VS Code integration docs: code.claude.com/docs/en/vs-code
  • Bash tool reference: platform.claude.com/docs/en/agents-and-tools/tool-use/bash-tool

1.3 Replit Agent

Agent 3 (September 2026) can run up to 200 minutes autonomously and surfaces progress through a chat-style activity feed plus a checkpoint timeline. Each checkpoint is a savepoint the agent creates between tasks so users can roll back without losing work. While building, Agent 3 spins up a headless browser to test its own work and replays those browser sessions back into the activity feed (so you can literally watch it click around your app). Mobile and web home pages get live progress bars, fix-log entries, and push notifications when a run finishes.

  • Agent 3 launch post: blog.replit.com/introducing-agent-3-our-most-autonomous-agent-yet
  • Replit Agent docs: docs.replit.com/replitai/agent
  • Deep dive on 200-minute runs: theunwindai.com/p/replit-agent-runs-autonomously-for-200-mins-non-stop

1.4 Devin

Devin is the canonical "watch the agent work" product. Its Workspace is a four-tab pane (Shell, Browser, Editor, Planner) that the user can either switch manually or set to "Following" mode, where the active tab auto-switches to whichever environment Devin is using. A time slider at the bottom of the Workspace lets users scrub backward and forward through the full session; the chat pane re-syncs to whichever message was active at that timestamp. The Planner tab holds a checkable plan Devin produces before coding, which users can edit or reorder before approving.

  • Devin intro: docs.devin.ai/get-started/devin-intro
  • Interactive Planning docs: docs.devin.ai/work-with-devin/interactive-planning
  • In-depth UI breakdown: ppaolo.substack.com/p/in-depth-product-analysis-devin-cognition-labs
  • Devin 2.0 update (proactive planning): cognition.ai/blog/devin-2

1.5 Lovable

In Agent Mode, Lovable shows "visible tasks" that stream during a build: the current step, files being modified, and tools being used (search, web fetch, image generation). The live preview re-renders as code lands, and a Details view exposes file diffs and per-message summaries. Above the chat input sits a prompt queue (up to 50 items): users can reorder, edit, pause, or remove queued prompts while a build is in progress. Lovable processes one task at a time, so the queue is the only way to "batch" follow-ups.

  • Agent Mode docs: docs.lovable.dev/features/agent-mode
  • Agent Mode launch blog: lovable.dev/blog/agent-mode-beta
  • FAQ on visibility: lovable.dev/faq/ai-agent/agent-mode

1.6 n8n

n8n splits the live and historical views. While a workflow runs, nodes light up green, red, or grey on the canvas in real time; after the fact, the Executions list (a per-workflow tab plus a global "All Executions" view) logs every run with status, start time, and duration. Users can filter by status (success, error, running, waiting), workflow, time range, and custom data attached in a Code node. Clicking a run reopens the canvas in its exact end state with per-node input and output, and a failed node shows the error, the input data at the time of failure, and a stack trace. The catch is retention: execution data prunes on an env-var schedule (EXECUTIONS_DATA_MAX_AGE), and self-hosted users routinely lose history sooner than they expect.

  • Execution history walkthrough: rapidevelopers.com/n8n-tutorial/how-to-view-execution-history-in-n8n
  • Executions docs: docs.n8n.io/workflows/executions
  • All executions view: docs.n8n.io/workflows/executions/all-executions

1.7 Zapier

Zapier's Zap history page lists every run on the "Zap runs" tab (10 per page) with a status badge for success or failure. Selecting a run shows the Zap version used and the data each step received and sent. Users can filter by app, folder, owner, status, and date, and search by version number or name. Retention is capped at 60 days and 10,000 runs, with export as the only way to keep more, so anything older silently falls off.

  • View and manage Zap history: help.zapier.com/.../View-and-manage-your-Zap-history
  • View specific run details: help.zapier.com/.../View-specific-Zap-run-details
  • Export Zap history: help.zapier.com/.../Export-your-Zap-history

1.8 Make

Make's scenario history logs each run with date and time, status (success, warning, error), duration, operations consumed, and data transferred, plus a separate change log of who edited the scenario. Opening a run's detail reopens the scenario canvas with markers showing how far execution reached; clicking any module shows its input and output bundles for that run. Make also surfaces a Run ID column whose UUID-style identifiers drew a wave of user pushback in 2025 (covered in Theme 8).

  • Scenario history: help.make.com/scenario-history
  • Scenario execution history: help.make.com/scenario-execution-history

Part 2 — User pain points (themed insights)

Five themes emerged from Reddit, GitHub Issues, the Cursor and Replit community forums, Hacker News, and product changelog comments. Sources were mined for verbatim quotes; thin spots are flagged.

Theme 1: The stream is too noisy by default

Users want to see what's happening, but the default density buries the signal in reasoning text and verbose tool output.

  • Claude Code (VS Code): "The VS Code extension currently displays extended thinking/reasoning blocks inline during responses. There is no way to suppress or collapse these automatically. For users who do not need to see the model's reasoning process, this creates significant visual noise and makes responses harder to read." (Issue #52640)
  • Cursor: Anthropic publicly traced an April 2026 Claude Code quality complaint wave to a "reduce verbosity" prompt change that backfired on coding quality and had to be reverted four days later (Anthropic postmortem).
  • Cursor 2.0 shipped a "compact / balanced / detailed" tool call density toggle in direct response to this kind of feedback (Cursor changelog).

The pattern: products that ship verbose-by-default end up retrofitting a density control. Worth considering for AIRO from day one rather than after the fact.

Theme 2: When the stream goes quiet, users can't tell if it's working or frozen

The opposite failure mode: when the agent buffers output or hits a long-running step, users have no signal whether progress is happening.

  • Claude Code: "When Claude runs bash commands in the VS Code extension's graphical panel, the output is buffered and only displayed after the command completes. This makes it difficult to monitor progress of long-running commands, know if a command is stuck or still working, see incremental output that might inform whether to cancel early." (Issue #14280)
  • Claude Code (same issue): "Currently I see nothing until it finishes. No way to know if it's working or frozen."
  • Replit Agent 3: "My main issue is that the agent just STOPS. No error, no crash. Just POOF stops working." (mikecoffin11, Replit forum)
  • Replit Agent 3: "When I see the message 'Agent optimizing memory…' I know I'm about to have a bad day." (mikecoffin11, same thread)
  • Replit Agent 3 workaround: "I started manually refreshing/reloading the app. After each refresh the AI would go to 'Thinking…' and spit out its report." (capacitron, same thread)

A streaming UI is only useful if it streams. Buffered output and silent stalls undermine the whole point of letting users watch.

Theme 3: The planning surface looks great in the demo but is fragile in practice

The todo-list-style planner is now table stakes, but multiple products ship it half-broken.

  • Cursor: "The To-Do lists panel doesn't show up and no tasks are created, even though the To-Do List feature is enabled in settings." (Cursor forum)
  • Cursor (same thread): "I'm being asked to burn more tokens to feign complexity to invoke the tool." User had to pad the prompt to trick the agent into calling the todo tool.
  • Cursor: "Plan mode is struggling to update the ToDo list after the plan builder produces the initial plan, even when changes are agreed upon with the agent." (Cursor forum)
  • Devin: The Planner pane is well-received but underused. One reviewer noted it functions as a "straightforward to-do list" but is potentially "under-leveraged for managing complex requirements" (Substack analysis).

A plan that disappears, doesn't update, or only appears on certain prompts is worse than no plan at all. Users notice when planning is theater.

Theme 4: Long-running agents need a way to look backward, not just forward

Once a session runs more than a few minutes, the linear chat scroll stops being enough. Users need to revisit what happened earlier.

  • Devin: The time slider is the strongest example. "While navigating in time, the chat is automatically scrolled to the messages exchanged before triggering the activities currently displayed." (Substack analysis)
  • Replit Agent 3: Checkpoint-as-rollback is the dominant scrubbing mechanism. The community fix for stuck loops is "use the Rollback feature to revert to a previous checkpoint before the loop started." (Replit forum)
  • Claude Code: No scrubbing primitive at all. The conversation is the timeline, which becomes unwieldy in sessions with hundreds of tool calls (Stella Laurenzo's audit covered 234,760 tool calls across ~6,800 sessions, per InfoQ).

For AIRO recipes (which can have many steps), some scrubbing primitive (checkpoints, step list, time slider) starts to matter once session length crosses a threshold. Worth investigating where that threshold is for automation builds.

Theme 5: Cost transparency is now part of the activity UI

In 2026, users expect the activity timeline to also surface what each step costs them. When it doesn't, trust erodes fast.

  • Replit Agent 3: "Charged me 31c to think but didn't tell me what it found out. Then charged me 7c to write." (MarcusTettmar1, Replit forum)
  • Replit Agent 3: "30 bucks a prompt sometimes is wild." (joliver2, same thread)
  • Replit Agent 3 (Reddit): One user reported being charged $350 in a single day under dynamic pricing (summary in Replit Agent Review).
  • Lovable: Cost-per-message is exposed in a three-dot menu on each message, which is one of the cleaner patterns observed (Lovable docs).

If AIRO charges per build or per credit, surfacing cost inside the activity stream (not in a separate billing tab) is worth scoping early.

Theme 6: What users actually want is "Show me AND let me steer"

This is the deeper pattern under themes 1 through 5. The activity timeline isn't just a status display. Users want to do things with it.

6a. Filter and collapse mid-stream. Hide thinking when they trust the agent, expand it when they don't. Cursor's compact-balanced-detailed toggle is the cleanest version of this so far.

For AIRO: a global "verbosity" control on the activity panel could let power users hide reasoning and surface only file edits and tool calls.

6b. Pause, reorder, redirect. Lovable's prompt queue is the strongest example: users can reorder, edit, pause, or remove queued prompts while a build is in progress.

For AIRO: if recipes have multiple build phases, letting users queue follow-up tweaks ("after this works, also add error handling") without breaking the current run could be worth considering.

6c. Scrub backward without losing forward. Devin's time slider syncs the chat to whatever timestamp you scrub to.

For AIRO: probably overkill at v1, but for long recipe builds, the ability to revisit "what did the agent do at step 4 again?" without re-reading 30 messages is the underlying need.

6d. Cancel early when you see it going wrong. Claude Code users complaining about buffered bash output cited this directly: "If I notice an error mid-stream, I could cancel early instead of waiting for completion."

For AIRO: a real-time stream that can't be cancelled mid-step is just a slow log. The "interrupt mid-action" affordance is the load-bearing part.

Themes 1 through 6 came mostly from AI coding agents, where the live stream is the main surface. The workflow-automation tier (n8n, Zapier, Make) surfaces a different cluster of pain, centered on the run-history surface. That surface matters more for AIRO than for a coding agent, because automation runs are recurring and often unattended, so the user's first look is usually after the fact, not live.

Theme 7: Run history disappears without warning, and there is no way to search what is left

Retention limits are opaque and almost always discovered post-incident, when the user goes looking for the run that broke and it is gone.

  • n8n: "All my 100k executions disappeared out of nowhere. Where did they go and how can I retrieve them and prevent this from happening again?" (shrjrwkls, n8n community)
  • n8n: "Execution history is being aggressively pruned after approximately 13 hours, despite having configured a much longer retention period." (n8n_addict, n8n community)
  • Zapier: "I can't check my task history more than one month old." (ComTH, Zapier community)
  • And once the history is there, you cannot search it. Activepieces: "the only way to find out what happened is to go through each execution history manually." (Matthew_James, ~1,000 executions/day, Activepieces community). n8n: "It seems that we ONLY list the last 20 runs and you may not scroll." (n8n community)

For AIRO: retention should be explicit and visible, not a silent gap. A "history older than [date] will be pruned" signal, plus search by trigger value, input, or outcome, is the difference between a usable log and scroll-to-find that breaks at production volume.

Theme 8: The run log speaks developer, not business user

When the history UI exposes technical implementation detail, the people who actually run automations cannot read it. Make shipped a Run ID column in 2025 and four separate users pushed back on it within 24 hours.

  • "I've been pulling my hair out over that new Run ID column. Long hash strings like 'ae5e8442c31349afbd9019…' taking prime real estate. The interface looks like a developer debug panel." (Ram_Moshkovits, Make.com community)
  • "Wall of meaningless hash characters that I have to mentally filter out. The Run ID serves zero purpose in day-to-day scenario management." (Gilad_Scher, same thread)
  • "Execution ID as main run name is driving me crazy too. Exposing technical implementation details to end users." (aderet_trachtingot, same thread)

For AIRO: the audience is business users, so this is sharper than for a coding tool. Name each run by what triggered it (recipe plus a short trigger summary plus timestamp), not an execution UUID. The run identifier should answer "which run was this?" at a glance.

Theme 9: Rerun does not replay the actual failure

Every product ships some form of rerun, and in every product it reruns the wrong thing: stale data, partial scope, or no ability to pick the one case that failed.

  • Zapier: "Zapier can't run failed Zaps from the initial step to the last step. This feature is half baked... The failed Zap is attempting to rerun again with the old data that was wrong." (ZapUser0001, Zapier community)
  • Make: "If 8 cases failed, I cannot play the 5th one only it seems. Is there any way to decide which one to execute?" (Gaurav_Srivastava, Make.com community)
  • Make: "It is hard to test when you can not run on sample data/rerun on existing runs." (Jason M., G2, Make.com)

For AIRO: rerun is only useful if it replays the exact conditions of the original failure. Replay from a chosen step with the original input, and consider selective replay of one item from a batch rather than all-or-nothing.

Theme 10: The log shows what ran, not why the agent decided it

This is the gap specific to AI agent timelines versus traditional workflow logs. A workflow log records the steps. An agent log needs to record the choice.

  • "Logs capture what happened, not why. Standard logging cannot answer whether an agent should have taken an action, only that it did." (TierZero)
  • "The rollback log says 'deployment X reverted.' It does not say why." (TierZero)
  • "68% of organizations can't tell if a production action was taken by a human or an AI agent after the fact." (TierZero)

For AIRO: capturing decision context (why this branch, why this connection, why this tool) rather than just tool calls is what separates a purpose-built agent timeline from a ported workflow log. It also lands close to Workato's audit heritage, so it is worth treating as a differentiator rather than a nice-to-have.

The biggest gap surfaced by this research

Every product researched treats the activity timeline as a display. The strongest ones (Lovable's queue, Devin's slider, Cursor's plan mode) also treat it as a control surface where users can steer the in-flight agent. For AIRO, the design question worth pressuring is: when users see something they don't like in the activity stream, what can they do about it without cancelling the whole run? That answer probably shapes the panel more than the visual layout does.

Adding the workflow-automation tier surfaced a second gap. Every product's log answers "what ran." None answers "why the agent decided that." For a coding agent the steps are usually enough, but for a Workato product with deep audit heritage, an agent timeline that captures decision context (why this branch, why this connection, why this tool) rather than just a list of tool calls is a credible right-to-win. It is also the cleanest line between a purpose-built agent timeline and a workflow execution log ported forward. Worth pressuring early, because it likely shapes what each step has to record from day one.

Part 4 — Research grounding

A few papers and pattern references back the design direction, and they answer the "is showing the work just trust theater?" question directly.

  • Interactive Reasoning ("Hippo"), ACM IUI 2025 (arXiv 2506.23678). Visualizes chain-of-thought as an editable hierarchy. In a study of 16 users, surfacing structured reasoning let people catch erroneous generations early and steer the model mid-run. This is the strongest evidence that exposing the agent's work is not decoration. The measured payoff is earlier error-catching and intervention, which argues for an interactive activity panel, not just an observable one. It lines up with Theme 6 (show me and let me steer).
  • Shape of AI, "Stream of Thought" pattern (shapeof.ai). The named pattern for this exact surface: plan plus execution log plus reasoning summary, with each step in an explicit state (queued, running, waiting for approval, error, completed) and an editable plan shown before execution. Usable design vocabulary for the AIRO panel, and it reframes the spinner as the fallback rather than the main event.
  • Wait-time baselines. NN/G's progress indicator and response-time work sets the rule that past 10 seconds you need determinate progress plus an interrupt. Agent runs almost always exceed that, so by the existing rule they require real progress, not a bare spinner. Microsoft's UX Design for Agents is the first-party anchor for "agent status is clearly visible at all times," with progressive disclosure of reasoning to calibrate trust. Both map onto Themes 1 and 2.
