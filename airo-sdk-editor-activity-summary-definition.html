AIRO code agent: the activity summary

A definition doc for the live activity summary (the "stream of thoughts") shown while Arrow edits connector code in the SDK editor. Problem, principles, and the design alternatives on the table. Research-stage, before constraints lock. PMO-2980.

On this page 1. The real problem and the goal 2. Principles we want to follow 3. Design alternatives (pros, cons, who does it)

1. The real problem and the goal

When Arrow edits a connector, the user has to be able to follow what it is doing to their code. Today the panel says one thing while it works: "Arrow is thinking." That is the starting point we are replacing, and it is the reason this is worth defining carefully rather than copying a coding tool's stream wholesale.

The real problem

A connector developer watching Arrow work needs to answer two questions, continuously, and the current UI answers neither:

  • "What is it doing to my code right now?" A blank "thinking" spinner gives no content. The opposite move, dumping the full reasoning and tool stream inline the way Claude Code does by default, over-answers it and buries the signal in noise (filed by users as "significant visual noise" with no way to collapse, Claude Code #52640).
  • "Is this going the right way, or do I stop it?" This is the one that actually matters, and it is the hardest. By the time a wrong edit has finished landing in a 1,000-line connector, the cheap moment to intervene has passed.

It matters more for this product than for a general coding agent for three concrete reasons. Connectors reach 1,000+ lines, so a wrong edit is easy to miss. Arrow writes diffs directly into the working file, so a bad turn has immediate consequences. And the audience is a developer who owns the connector and will be blamed if it breaks, so the tolerance for "I didn't see that change happen" is low. The two failure modes from the activity timeline research bracket the design: too noisy (the stream buries the signal) and too quiet (the user reads silence as a crash, like Replit Agent 3 users describing it "just POOF" stopping, Replit forum).

The goal

An activity summary that is:

  • Glanceable. One line of intent per step, in plain language, so the user understands what is happening without reading a transcript.
  • Interruptible at the cheapest moment. The user can stop mid-step the instant a wrong direction shows, not only after the edit lands.
  • Trustworthy enough to look away. The summary and the resulting diff are good enough to review afterward, so the user does not have to babysit the stream.

In one line: the activity summary is a control surface, not a log. That reframe (from the research's "biggest gap" finding) shapes the panel more than the visual layout does.

Non-goals

  • A full reasoning transcript. Reasoning is available on demand, not the default.
  • Re-rendering the connector code. The editor already shows the code and the diff; the activity summary narrates the process happening to it, not the blueprint.
  • Post-run execution history and debugging. That is a separate surface with its own retention and search problems.

2. Principles we want to follow

Six principles, each grounded in how Cursor, Claude Code, and others got it right or wrong.

1.Intent before narration.

Announce what each step is about to do in one line, before or as the edit lands ("Editing connection auth", "Adding get_user action"). This is what lets the user catch a wrong direction at the cheapest point. The SDK backend tools (getLatestConnectorCode, validateCode, writeLatestCode) already map to readable step verbs, so the vocabulary is close to free.

Pattern: Shape of AI, Stream of Thought. Closest shipped example: Cursor Plan Mode states the plan before writing code.

2.One default density, with an escape hatch in both directions.

Verbosity is not "too many words," it is "no chosen default." Default to one line per action, collapse reasoning, and let users expand for detail or dial it down further. Ship the collapse from day one rather than retrofitting it.

Claude Code shipped inline thinking with no toggle and users called it noise (#52640). Cursor retrofitted a compact / balanced / detailed control (Cursor 2.0). Anthropic's own "reduce verbosity" change backfired on quality and was reverted in four days (April 23 postmortem), so the answer is user control, not a fixed terse default.

3.The change summary is computed from the diff, not the model's claim.

Derive each summary line from the actual computed diff (which block changed, lines added and removed) and let the model only phrase it. A summary that cannot disagree with the editor is the feature.

In the diff lifecycle research, users in three products caught the agent claiming changes that never happened. NaturalEdit (arXiv 2510.04494) found summaries help most when clickably linked to the code they describe, which also doubles as long-file navigation.

4.Always interruptible, mid-step, with a heartbeat.

A persistent stop that halts mid-step, plus a live current-action line so silence never reads as frozen. A stream you cannot cancel mid-step is just a slow log.

Claude Code users asked to cancel early on buffered long-running output (#14280). The interrupt-mid-action affordance is the load-bearing part of the activity timeline research's "show me AND let me steer" theme.

5.Earn the right to look away.

The point of a trustworthy summary is that the user does not have to watch. A clean per-turn summary plus a reviewable highlighted diff means the user can disengage and come back, instead of monitoring every token.

Microsoft, UX Design for Agents (agent status visible at all times, progressive disclosure of reasoning). Past 10 seconds of wait, determinate progress plus an interrupt is the baseline (NN/g response-time limits).

6.Narrate the process, show the code in the editor.

The activity summary's job is the steps, handoffs, tools, and status, not a second rendering of the connector. The editor owns the code and the diff; the summary owns the story of what is being done to it.

Carried over from the AIRO working-view principle: the "agent working" view shows the multi-agent process, not the recipe or connector blueprint.

3. Design alternatives

Five ways to display the live activity, from thinnest to richest. Each has pros, cons, who ships it, and where it would land for AIRO. These are not mutually exclusive; the realistic path is to build a strong core and grow it.

A. Raw streaming transcript Avoid as default

Stream reasoning text and full tool input/output inline, in order, as it happens.

ProsCons
  • Maximum transparency, nothing hidden
  • Cheapest to build (just render the event stream)
  • Power users can see exactly what the model did
  • Buries intent in noise; hard to scan
  • The signal a user needs ("what changed, is it right") is drowned out
  • Long sessions become an unscrollable wall

Seen in: Claude Code default. Rationale: users filed the inline thinking as visual noise with no collapse (#52640). Fine as the expanded detail view behind a toggle, wrong as the default surface.

B. Rolling one-line status only Too thin

A single current-action line with a spinner, e.g. "Arrow is editing connection auth…", replaced as it moves on.

ProsCons
  • Zero noise, very glanceable
  • Trivial to build; a direct upgrade to today's "Arrow is thinking"
  • Satisfies the heartbeat principle on its own
  • No history; the user cannot see what already happened this turn
  • No per-step record to review or to anchor a stop decision
  • If it stalls on one step, still reads as frozen

Seen in: most chat assistants' "typing…" state. Rationale: it is the smallest step up from the current failure, but it only answers "right now," not "what did it just do," so it cannot carry the control-surface goal alone. Useful as the always-visible heartbeat layer on top of option C or D.

C. Collapsed step list with status badges Strong core

An ordered list of steps, each a verb plus target block plus an explicit state (queued, running, done, error). Reasoning hidden behind a per-step expander.

ProsCons
  • Scannable and intent-forward; reads like a checklist
  • Each step is an intervention point, which serves "stop mid-step"
  • Maps cleanly onto the SDK tool calls
  • Completed steps collapse, so long sessions stay compact
  • Needs the backend to emit structured step events, not just a token stream
  • Vague labels ("Working…") would waste the format
  • Planning surfaces are fragile in practice and must be reliable to be trusted

Seen in: Shape of AI, Stream of Thought; Devin's Planner; Lovable visible tasks. Rationale: Cursor's todo list sometimes fails to appear or update, and users notice when planning is theater (Cursor forum); Devin's planner is well-built but reviewers call it under-leveraged (analysis). The structure is right; the risk is reliability, not concept.

D. Step list plus per-turn change summary linked to the diff Target direction

Option C while it works, then on completion a one-line summary derived from the diff ("Added get_user action, updated auth. +34 / −6"), where each summary line jumps to its hunk in the editor.

ProsCons
  • Digestible live, reviewable after; supports looking away
  • The diff-derived summary satisfies the kickoff should-have honestly
  • Clickable summary lines double as long-file navigation (the 1,000-line problem)
  • One feature covers three needs: summary, trust, navigation
  • More to build than C, and two surfaces (live steps, final summary) to keep coherent
  • Summary-from-diff requires the diff computed where the summary is generated

Seen in: Lovable's per-message summaries; the linking idea from NaturalEdit. Rationale: the diff lifecycle research names "no product ships a summary guaranteed to match the code" as the biggest open gap, so this is where AIRO can lead rather than catch up.

E. Full "watch it work" workspace with a scrubber Later

A multi-pane workspace (editor, validation, browser) with a time slider to scrub backward and forward through the whole session.

ProsCons
  • Powerful for long, autonomous, multi-environment runs
  • Strong "look backward" support without re-reading the chat
  • Overkill for v1 single-file connector editing
  • High build cost; the connector session is short next to Devin-scale runs
  • Risks the same under-leveraged fate when the task does not need it

Seen in: Devin's Workspace and time slider (200-minute autonomous runs). Rationale: the activity timeline research flags scrubbing as probably overkill at v1; the lighter version of the same need is "jump to step N," which option D already gives via linked summary lines. Worth revisiting only if connector sessions grow long enough that the linear list stops being enough.

Where this points

The realistic path is to build option C as the core and grow it into D, with the option B status line riding on top as the always-visible heartbeat. Option A becomes the expanded detail view behind a toggle, not the default. Option E is a later bet that only pays off if sessions get long. The open question worth pressuring with the team, straight from the research: when a user sees something they dislike in the stream, what exactly can they do about it without cancelling the whole run? That answer shapes the panel more than the layout does.
