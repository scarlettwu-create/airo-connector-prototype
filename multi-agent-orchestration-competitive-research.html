Multi-Agent Orchestration — Competitive Research

Context. For AIRO's agentic surface at Workato, looking at how other AI products let one task run across several agents. Two use cases drive this: (1) splitting a task across multiple agents to finish faster (parallel decomposition), and (2) different agents playing different roles like writer, reviewer, or researcher. Feeds the "borrow proven patterns from leading AI products" principle the AIRO team already designs against.

Products covered. MorphMind, Claude Code (subagents), Devin / MultiDevin, CrewAI, Manus, Flowith, Ottogrid / Paradigm, Genspark Super Agent.

Scope. How each product orchestrates multiple agents on a single job: how work is split, whether agents have roles, what users see while it runs, what it costs, and where it breaks. Deliberately skews toward newer, more experimental products rather than the enterprise automation incumbents.

Part 1 — Orchestration behavior per product

1.1 MorphMind

Frames orchestration as a team of role-based specialists you can inspect and steer (tagline "Controllable AI agents, work you can stand behind"), with WorldSeed running agents that propose hypotheses, run experiments, and peer-review each other's work, Caliper attaching a calibrated trust score to every result, and Flowtrace turning a session into a reusable flow rather than a throwaway chat; the official copy is about role-based collaboration and verification, not parallel speed.

1.2 Claude Code (subagents)

Subagents are role-based assistants defined in markdown, each with its own prompt, tool allowlist, and model, that the main agent delegates to in isolated context windows; the main agent can spawn several in parallel, and an experimental agent teams mode lets full sessions coordinate through a shared task list and message each other.

1.3 Devin / MultiDevin

A coordinator Devin scopes a large task, spins up to 10 worker Devins that run in parallel in isolated VMs, and merges only the successful workers into one branch or PR (Devin can now manage Devins, March 2026); the workers are identical general-purpose Devins differentiated only by their assigned task, so there are no preset roles, and Cognition recommends it only for repeated, isolated work like lint fixes, migrations, and refactors.

1.4 CrewAI

Crews are role-based agents, each given a role, goal, and backstory, that run either sequentially or hierarchically under a manager agent that delegates and validates outputs, with independent tasks marked async_execution running in parallel and downstream tasks waiting on them.

1.5 Manus (Wide Research)

Wide Research is deliberately role-free parallel decomposition: a controller breaks a task into independent subtasks and spins up many subagents (the docs say hundreds), each a full Manus instance on its own virtual machine with a fresh context window, then aggregates the results into one output; Manus explicitly contrasts this with role-based manager/coder/designer systems.

1.6 Flowith

A 2D infinite agentic canvas where prompts and responses are branchable nodes across multiple models, with an orchestrator (Oracle) that breaks a task into editable steps and picks a tool per step, plus an autonomous execution engine, Agent Neo, that runs long multi-step jobs in the cloud; the parallelism is the multi-thread canvas plus Oracle's subtask split rather than distinct role agents.

1.7 Ottogrid / Paradigm (agents-as-a-grid)

Both put a parallel agent swarm behind a spreadsheet: in Ottogrid each column is an agent with its own prompt and tools and rows fill in parallel across hundreds of jobs, and Paradigm markets an agent in every cell with thousands of agents fanning out across the web to fill rows in parallel (founders cite roughly 5,000 cells a minute); role specialization is light, just the per-column prompt.

1.8 Genspark Super Agent

Super Agent runs a mixture-of-agents architecture combining nine LLMs and 80+ tools, decomposing a request and routing each subtask to the best-suited model (creative work to one, reasoning to another), then reconciling the outputs into one result; this is both parallel decomposition and strength-based routing across models.

A quick read on the two use cases: only Claude Code, CrewAI, and Genspark ship first-class support for both parallel and role-based out of the box. Manus, Flowith, and Ottogrid/Paradigm are parallel-first with little or no role concept. MorphMind is the clearest role-first bet, leading with peer review and trust scores rather than speed.

Part 2 — User pain points (themed insights)

Six themes, mined from the Anthropic and Cognition engineering essays, Hacker News, the CrewAI and AutoGen issue trackers, app store and Product Hunt reviews, and independent review blogs. One thing to flag up front: the complaint volume clusters on metered or autonomous products (Manus, Genspark) and developer frameworks (CrewAI, Claude Code, AutoGen). The newest role-based products (MorphMind, Paradigm, Ottogrid) have almost no public user complaints yet, partly because they are too new and partly because they market directly against the pains below. That absence is a finding, not validation.

Theme 1: The token bill scales with the agent count, and it lands as a surprise

The cost of fanning out is real and users discover it after the fact, not before.

  • Anthropic concedes the multiple in its own engineering post: "agents typically use about 4× more tokens than chat interactions, and multi-agent systems use about 15× more tokens than chats" (Anthropic Engineering, June 2025).
  • Genspark: "On my second day after subscribing to the Genspark Plus plan for $24.99 a month, I realized that my 10,000 credits had hit zero," because Super Agent "consumes around 100 credits per action" and runs by default for any prompt (satoshi hayasaki, note.com, March 2026).
  • Manus, from a five-star reviewer: "New credit feature is expensive. A single question could use up all your credits for a month!" (Apple App Store, April 2025).
  • CrewAI: "A crew of 5 agents processing a complex task can easily 10x your API costs compared to a single well-prompted agent" (Hugh McInnis, AgentRank, March 2026).

The pattern holds whether the meter is tokens (Anthropic, CrewAI) or opaque credits (Manus, Genspark). Credit-based products are worse because the user can't map "100 credits" to anything before they spend it. (An often-cited $47,000 Claude Code subagent run from aicosts.ai circulates as illustration, but the company is unnamed, so treat it as color rather than evidence.)

For AIRO: in a metered enterprise product, the cost of spawning N agents is worth showing before the fan-out, not just on the invoice. A pre-run estimate and a live running cost per agent would address the single most common complaint in this space.

Theme 2: Parallel agents step on each other, because they can't see each other's work

When agents run in parallel without shared context, their outputs don't fit together.

  • Cognition's case against multi-agent leads with this: a subagent asked to build part of a Flappy Bird clone "started building a background that looks like Super Mario Bros," so the pieces don't match, because "the decision-making ends up being too dispersed and context isn't able to be shared thoroughly enough between the agents" (Walden Yan, Cognition, June 2025).
  • On Hacker News: "My biggest hesitation about this is being stuck in merge hell. Even a minute or two needing to deal with that could negate the benefits of agents working in parallel" (babyshake, 2025).
  • The same thread surfaces a context limit, not just a token limit: "agents start losing track of their original goals around the 50k token mark" (pbd, 2025).

The strongest evidence is that Claude Code shipped a whole git worktree feature to keep parallel agents in separate working directories. It exists specifically to prevent two agents editing the same file at once and overwriting each other. When a product builds dedicated machinery to stop a failure mode, the failure mode is real.

For AIRO: if two agents can touch the same recipe, connection, or shared resource at once, the conflict has to be designed for, not discovered. Worth deciding early whether parallel agents get isolated scratch space or share live state.

Theme 3: You can't tell what each agent did, or which one broke

Once work is spread across agents, attribution and debugging get hard.

  • CrewAI: "When a crew produces bad output, figuring out which agent screwed up and why is genuinely difficult," and "the open-source version gives you limited visibility into inter-agent communication. You end up adding print statements like it's 2005" (AgentRank, March 2026).
  • Anthropic names the root cause: "Agents make dynamic decisions and are non-deterministic between runs, even with identical prompts. This makes debugging harder" (Anthropic Engineering, June 2025).
  • Manus, on the lack of any signal during a long autonomous run: "Manus does not warn when it is approaching its context limit. There is no yellow light. There is no 'I'm getting close to my limit, let me generate a handover log'" (Iolanda Fonseca, The Rio Times, March 2026).

This is the clearest open UX gap, and it's exactly where MorphMind is placing its bet: WorldSeed makes the agents' work inspectable and Caliper attaches a trust score to each result, both aimed at "see what the AI did" rather than "trust the black box." Nobody in the set has a strong unified view of what each agent did across a parallel run.

For AIRO: a single timeline or trace that shows which agent did what, in what order, with what result, would directly serve the observability gap and AIRO's existing run-history work.

Theme 4: The reviewer just agrees with the writer

The role-based use case has its own failure: adding a reviewer or critic agent doesn't guarantee the scrutiny it promises.

  • Research on multi-agent debate finds that "sycophancy can collapse debates into premature consensus, potentially undermining the benefits of multi-agent debate" (Yao et al., arXiv 2509.23055, 2025). The reviewer tends to agree rather than push back.
  • Practitioners reach the same conclusion about role crews generally: "for 70% of use cases, a single well-prompted agent with good tools outperforms a crew of mediocre agents" (AgentRank, March 2026).

The products that take use case 2 seriously are the ones engineering against rubber-stamping: MorphMind's WorldSeed has agents peer-review each other, and the cleanest reference for a genuinely adversarial reviewer is Google Jules's separate critic agent in the coding space. A writer-then-reviewer crew where the reviewer defaults to agreement adds cost without adding the check.

For AIRO: a reviewer role is only worth its cost if it actually disagrees sometimes. Worth defining what the reviewer is adversarial about and how a disagreement surfaces to the user, rather than assuming a second agent improves quality by existing.

Theme 5: The coordinator can't delegate cleanly, so agents loop or stall

The manager-and-workers pattern fails in predictable ways: agents hand work back and forth, or get stuck.

  • CrewAI: when multiple agents can delegate to each other, they fall into loops where A delegates to B and B delegates back to A (GitHub #1355; community thread).
  • AutoGen's group chat hits infinite loops because the framework uses brittle rules to decide when a conversation should terminate, fixed only by hard message caps (microsoft/autogen #133).
  • It shows up in autonomous consumer products too. Manus "will waste your credits over and over and over again, trying to achieve the same command that you gave it to begin with" (joblessgambler, Apple App Store, August 2025), and a Genspark user reports a task "constantly interrupted, and the operation must be restarted every 10 minutes" (Trustpilot, reviewer unattributed).

For AIRO: any coordinator-driven orchestration needs a hard ceiling (max rounds, max delegations, max spend) and a clear terminal state, so a stuck crew fails visibly instead of looping on the meter.

Theme 6: Watching the swarm becomes its own full-time job

Parallel agents promise speed, but the supervision they require can eat the gain.

  • "When reviewing agentic generated code I can trust nothing and have to verify every assumption and that introduces a massive overhead" (munk-a, Hacker News, 2025).
  • "I'm not convinced there is any hope for a productive, long-term, burnout-free parallel agent workflow. Not while they need even the slightest amount of supervision/review" (grim_io, 2025).
  • And the trust cost when an agent fabricates progress: Manus "produced fabricated evidence, curl responses, file listings, and status checks that were formatted like real server output but described a state that did not exist on the server" (The Rio Times, March 2026).

The deeper point is that splitting one stream of work the user could follow into several parallel streams they can't multiplies the review burden at the same time as the throughput. Speed on the generation side can become a bottleneck on the human side.

For AIRO: the win from parallelism is real only if the review surface keeps up. Worth designing the "what came back and is it trustworthy" moment as carefully as the fan-out itself, so the user isn't left auditing several agents at once with no help.

Part 3 — AIRO design implications

Competitor pain point What this could mean for AIRO
Multi-agent runs cost 15x a single chat (Anthropic), and credit burn surprises users (Manus, Genspark) Worth surfacing a cost estimate before a fan-out and a live per-agent running cost during it, rather than only on the invoice
Parallel agents overwrite each other's work; Claude Code needed worktrees to prevent it Worth deciding early whether parallel agents get isolated scratch space or share live state on recipes, connections, and jobs
Can't tell which agent did what or which one failed (CrewAI, Anthropic non-determinism) Worth a unified trace showing each agent's work, order, and result, building on AIRO's existing run-history surface
No warning before a long autonomous run fails (Manus) Worth a health signal that distinguishes a long-but-working run from a stuck one, and a handoff log before a hard stop
Reviewer/critic agents rubber-stamp the writer (sycophancy research) If AIRO offers role-based agents, worth defining what the reviewer actually challenges and how a disagreement reaches the user
Coordinator agents loop, delegating back and forth forever (CrewAI, AutoGen) Worth a hard ceiling on rounds, delegations, and spend, plus an explicit terminal state when a crew gets stuck
Supervising several parallel agents becomes its own job (Hacker News) Worth designing the review-and-verify moment so a user isn't auditing many agents at once unaided
Most coding tasks have "fewer truly parallelizable tasks than research" (Anthropic), single agent wins for ~70% of cases (CrewAI review) Worth treating "should this fan out at all" as an explicit, visible decision rather than the default

The biggest gap surfaced by this research

The unsolved problem across every product is observability and trust across multiple agents at once. Every product can spawn agents. None of them gives the user a confident answer to "what did each agent do, what did it cost me, and can I trust the result" while the run is happening. CrewAI users add print statements, Manus gives no warning before it fails, and even Anthropic flags non-determinism as a debugging problem it hasn't fully solved.

This is exactly the space MorphMind is betting its whole product on (inspectable work via WorldSeed, a trust score via Caliper), which suggests the market sees it as the open question too. For AIRO, the strategic call is less "can we run agents in parallel" and more "can we make a parallel or role-based run as legible and trustworthy as a single agent's." That connects directly to AIRO's run-history and activity-timeline work, and it's where a workflow automation product with a real audit-trail heritage could lead rather than follow.

A related framing worth putting in front of the team: the two most credible labs published opposite conclusions in the same month. Anthropic (multi-agent research system) argues multi-agent wins for breadth-first, parallelizable work; Cognition (Don't Build Multi-Agents) argues to default to single-threaded agents because dispersed decisions produce incoherent output. Both are right within their domain. The design question for AIRO is which kind of work its users actually have, and the answer probably differs by task, which argues for making the fan-out a deliberate choice rather than an always-on behavior.

Part 4 — Theoretical framework: why multi-agent systems fail (MAST)

Source. Why Do Multi-Agent LLM Systems Fail?, Cemri et al. (UC Berkeley), arXiv 2503.13657 (March 2025).

The first empirically grounded failure taxonomy for multi-agent LLM systems, built from 1,600+ annotated traces across 7 frameworks with strong inter-annotator agreement (Cohen's kappa 0.88). Its headline finding is the useful one for a product team: most failures come from poor system design and orchestration, not from the models being too weak. Bigger models don't fix it; better structure does.

How the framework maps to AIRO

MAST sorts 14 failure modes into three categories, and each is a design surface AIRO controls:

MAST category What it covers AIRO design surface
Specification issues Ambiguous roles, unclear task specs, bad system design How clearly each agent's role and scope is defined when a crew is set up
Inter-agent misalignment Agents ignore each other, make conflicting assumptions, derail Shared context and conflict handling between parallel agents (Theme 2)
Task verification Weak or missing checks on the final output Whether a reviewer role actually verifies, or just rubber-stamps (Theme 4)

The "inter-agent misalignment" category is the academic version of Cognition's dispersed-decisions argument, and "task verification" is the academic version of the rubber-stamp reviewer problem. The two failure modes users complain about most are the two the research says matter most.

For shared vocabulary when speccing this, Microsoft's AI Agent Orchestration Patterns names the common topologies cleanly: sequential, concurrent, group chat, handoff, and magentic (a manager that plans on the fly). Useful as a neutral naming scheme so design and engineering mean the same thing by "parallel" or "handoff."

Constraints worth understanding

  • The taxonomy explains what goes wrong, not how to fix it. It points at design surfaces; it doesn't hand you the design.
  • It studies developer frameworks, not end-user products, so the orchestration failures are well covered but the supervision-burden and cost-surprise pains (Themes 1 and 6) come from user reports, not the paper.
  • "Better structure beats bigger models" is the load-bearing claim, and it's encouraging for a product team because the fixes are design decisions AIRO can make, not model capabilities it has to wait for.
  • No rigorous HCI study yet exists on the cognitive load of supervising several agents at once, which is the gap directly under Theme 6. Worth watching for, or worth AIRO generating its own evidence on.
