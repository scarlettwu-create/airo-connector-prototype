AIRO SDK editor: in-editor AI code insertion and diff — Competitive Research

Context. Research for the AIRO SDK editor migration (PMO-2980), where AIRO inserts generated code directly into the SDK editor at the correct lines within the relevant SDK blocks and shows the user a before-and-after comparison to accept or reject. This is the research-stage artifact, before constraints are locked.

Products covered. Cursor, GitHub Copilot Edits, Fivetran Connector SDK (with Claude in VS Code), Tray.ai (Claude CDK), Celigo (Copilot / Ora), Boomi (Agentstudio + Suggest).

Scope. How each product turns an AI suggestion into code that lands in a working file: whether it inserts at the right lines, how the before/after diff is shown, how accept/reject works, and what breaks when it goes wrong.

Part 1 — In-editor code insertion behavior per product

The six products split into two groups. Cursor, Copilot, and Fivetran do real in-editor insertion with a diff. Tray, Celigo, and Boomi do not, which is itself a finding worth holding onto for Part 3.

1.1 Cursor

Cursor inserts code at the target lines as an inline color-coded diff (red deletions, green additions); Cmd+K edits a selection with Cmd+Enter to accept and Cmd+Backspace to reject, while Composer/Agent mode applies edits across files with per-file and per-hunk Keep/Undo controls, though a recent build started applying agent edits straight to disk with no red/green diff and no accept/reject gate (inline edit docs, changelog, regression report).

1.2 GitHub Copilot Edits

Copilot Edits applies proposed changes directly into each file as an inline diff reviewed per chunk (Tab to accept, Alt+Delete to reject, F8 to jump to the next, a per-file checkbox, or Accept all) backed by a checkpoint/rollback system, though selecting Keep can re-apply stale or composite diffs and overwrite newer work (Copilot Edits in Visual Studio, review AI edits in VS Code, multi-file editing changelog).

1.3 Fivetran Connector SDK

Used with Claude in VS Code, the Connector SDK has the AI write connector.py, configuration.json, and requirements.txt directly into the project through the editor's agent edit-and-accept flow (it proposes a plan, then creates files in sequence that you accept in place), with a parallel terminal path using fivetran debug and CLI agents for local testing and iteration (Claude in VS Code tutorial, Claude Code developer workflow blog, csdk agent tools).

1.4 Tray.ai (Claude CDK)

Tray has no native in-app editor or diff panel: the Claude CDK runs Claude Code inside your own terminal or IDE pointed at an OpenAPI spec (tray-cdk connector import [OPENAPI_SPEC]) and scaffolds the connector project locally, so the before/after review is whatever Claude Code itself provides rather than a Tray surface (Tray Headless, Claude Code blog post, generate from OpenAPI docs).

1.5 Celigo (Copilot / Ora)

Celigo's assistant (Copilot, now extended by the Ora agent launched around April 2026) is an in-product conversational panel that scaffolds JavaScript hooks, SQL, and Handlebars, and stages every change as a reviewable draft you must explicitly approve (you can ask it to show exactly what will change first), which is closer to a change-summary preview than a line-level inline code diff (Copilot AI-driven resolution, Ora product page).

1.6 Boomi (Agentstudio + Suggest)

Boomi GPT and Agent Designer turn natural-language prompts into integrations and agents inside a visual no-code canvas with a Test Agent panel and a reasoning trace, and the older Boomi Suggest proposes field mappings in confidence buckets, so the user reviews and accepts suggested mappings or agent behavior in the visual builder rather than getting code inserted into an editor with a diff (Agent Designer overview, Agentstudio testing blog, Boomi Suggest).

Part 2 — User pain points (themed insights)

Six themes emerged. The richest evidence for the in-editor insertion mechanics comes from Cursor's forum, Hacker News, and GitHub Issues/Discussions. The iPaaS players (Tray, Celigo, Boomi, Fivetran) have less public complaint volume aimed squarely at AI code-gen because the features are recent, so their evidence clusters around the adjacent pain: debugging wrong output and the manual work the AI was supposed to remove.

Theme 1: The AI edits the wrong lines and clips the code around them

The single most common in-editor complaint is not that the generated code is bad, it's that it lands in the wrong place and damages working code next to it.

  • GitHub Copilot Edits: "it removes the closing bracket of the method. Some times it removes the first few lines in the next method." (2dcon, discussion #58416, Aug 31 2025)
  • GitHub Copilot Edits: "Copilot auto deletes unrelated code blocks during editing without any specific explanation." (itech0307, discussion #58416, Jul 9 2025)
  • Cursor: "it liberally deleted a chunk of code it must have thought was extraneous but at the same time introduced a huge privacy vulnerability." (neuralkoi, Ask HN thread, Mar 2025)
  • Cursor: "It has happened to me on very large files and when you've had a long composer session. I usually check every file that is being modified for large removals." (haffi112, same HN thread, Mar 2025)

This is the localization problem, and it is exactly what AIRO is taking on by promising to update code "at the correct lines within relevant SDK blocks." The benchmark evidence in Part 4 shows getting the insertion point right is genuinely hard, so it's worth treating insertion accuracy as a first-class requirement, not a side effect of good generation. For AIRO: scoping each edit to a known SDK block (rather than a free-floating line range) could narrow the surface where this failure happens.

Theme 2: The accept/reject control itself destroys good work

Across the two most mature diff products, the Keep/Undo controls are a top source of data loss. Accepting can re-apply stale edits over newer ones, and rejecting can wipe valid code.

  • GitHub Copilot Edits: "Selecting Keep re-applies stale Copilot diffs and overwrites newer changes. Selecting Undo reverts all Copilot-associated changes, deleting valid work already present on disk." (pramodsurya, discussion #183864, Jan 9 2026)
  • GitHub Copilot Edits: "older, superseded revisions of the code being reapplied, overwriting subsequent manual fixes or AI-driven revisions." (brylie-upcloud, vscode issue #265794, Sep 9 2025)
  • Cursor: outdated diffs appear for files that were not recently edited, with Cursor asking users to accept or undo changes on files they weren't working on (forum thread, late 2025)
  • GitHub Copilot Edits: "I keep having the same issue to the point where I would strongly prefer to completely disable 'pending changes from chat' and auto-approve any and all changes." (h0m3us3r, discussion #183864, Apr 19 2026)

The root cause in both products is that accept/reject replays edit history instead of comparing two fixed states. For AIRO: a two-state model (the original SDK code vs the proposed final code) per accept action, with a non-destructive reject, would sidestep the whole class of "the undo deleted my work" complaints. This is worth pinning down before the diff UI gets built, since it's an architecture decision more than a UI one.

Theme 3: Silent collateral edits, and the AI denies making them

When the edit touches code outside its intended scope, the trust damage is worse if nothing flags it.

  • GitHub Copilot Edits: "copilot even claims it didn't touch the unrelated code it has deleted. Incredibly dangerous." (astrobc1, discussion #58416, Aug 14 2025)
  • Cursor: "If you don't supervise, you will get snagged, and if you miss it and continue, it'll continue writing code under the assumption the deletion was fine." (tony, HN thread, Mar 2025)

The pattern users hate is a quiet change they didn't ask for, discovered later. The before/after comparison in the PMO-2980 description is the main defense here: if every line AIRO changed is visible and attributable, collateral edits stop being silent. For AIRO: worth making the diff show the full blast radius of a change, including lines AIRO touched outside the block the user asked about, rather than only the lines it meant to change.

Theme 4: Reviewing large or multi-file diffs strips the context needed to judge

Even when the edit is correct, a consolidated diff view that hides the surrounding code makes it hard to decide whether to accept.

  • Cursor: "The whole point of seeing where the changes are is to be able to accept a change, see how it fits." (Akhil_Gupta, diff view thread, Dec 3 2025)
  • Cursor: "I need to see more context when I'm evaluating a change." (Gabriel_Richards, same thread, Dec 3 2025)
  • Cursor: "It's KILLING my productivity... the agents keep getting stuck asking me to review things in panes." (F1LT3R, same thread, Dec 12 2025)

There's a real tension here: users want enough context to trust the change, but forced review panes interrupt flow. For AIRO: showing the diff inline within the SDK block (with the unchanged code around it still visible) tends to beat a separate summary pane for this kind of judgment. The volume of generated SDK code per request will decide how sharp this trade-off gets.

Theme 5: After generation, users can't tell why the output is wrong

This is the dominant theme on the iPaaS side. The generation step is fine; the problem is debugging what it produced when it doesn't work.

  • Fivetran: "Sometimes we get obscure or silent failures that require reaching out to Fivetran support. It's hard to tell which side the error is coming from." (Lucy W., G2, Apr 9 2026)
  • Fivetran: "Clearer root cause rather than generic error messages would help us debug or triage." (Luke M., G2, Apr 9 2026)
  • Celigo: "Some of the error messages doesn't have the proper details, which causing our users to spend more time investigating." (Bindu J., G2, Mar 9 2024)
  • Boomi: "It can be difficult to debug and handle errors, especially when dealing with complex operations." (tarun p., G2, Apr 7 2026)

PMO-2980's acceptance criteria already say the connector has to stay functional after AIRO edits it. That's the right instinct given this theme. For AIRO: the moment after an edit is accepted is where this pain lives, so it's worth deciding early how AIRO surfaces a broken-connector state and points at the line it came from, not just whether the edit applied.

Theme 6: The AI doesn't remove the manual code work it implied it would

Users across the iPaaS products report that the AI gets them partway, then hands back something that still needs real technical work.

  • Tray: "Custom API configurations and data manipulation require significant technical knowledge. The gap between 'simple automation' and 'production-grade enterprise workflow' demands more technical depth than the marketing implies." (Integrate.io review, 2026)
  • Celigo: "Some advanced use cases require a deeper technical understanding... working with complex APIs." (Nuri Vladimir E., G2, Feb 13 2026)
  • Boomi: Boomi Suggest produces maps "with 80 percent of the required mappings," and the docs tell users to "verify the accuracy of the AI-generated suggestions," so roughly a fifth of the mapping is still manual plus false positives on similarly named fields (Boomi Suggest)
  • Celigo: "I still 'dislike' the inaccuracy of some of the AI features." (Nathan C., G2, Mar 26 2026)

The honest read is that none of these tools fully closes the loop, and users feel the gap between the pitch and the leftover work. For AIRO: the copy-paste-reposition step PMO-2980 removes is real and worth removing, and being precise about what AIRO does not yet do (which SDK blocks it can edit reliably, which it can't) would set expectations better than these competitors have.

Part 3 — AIRO design implications

Competitor pain point What this could mean for AIRO
Copilot and Cursor edits clip closing brackets and delete lines of the next method (Theme 1) Worth scoping each AIRO edit to a known SDK block boundary so insertions can't run past the structure they belong to
Keep re-applies stale diffs and Undo deletes valid work (Theme 2) Worth modeling accept/reject as a comparison of two fixed states (original vs proposed) rather than a replay of edit history
AI denies touching code it silently changed (Theme 3) Worth making the before/after comparison show every line AIRO changed, including collateral edits outside the requested block
Consolidated diff panes hide surrounding context and stall review (Theme 4) Worth showing the diff inline within the SDK block with unchanged code still visible, rather than a separate summary pane
Silent failures and generic errors after generation (Theme 5) Worth deciding how AIRO surfaces a broken-connector state post-edit and links it back to the specific line
AI leaves ~20% of the work manual without saying so (Theme 6) Worth being explicit about which SDK blocks AIRO can edit reliably today and which still need hand-finishing
Cursor regression auto-applied edits to disk with no accept/reject gate (Part 1.1) Worth treating the accept/reject gate as a non-negotiable, since users notice immediately when it disappears
Tray pushes the diff experience out to Claude Code in the user's own IDE (Part 1.4) Worth keeping the review surface inside the SDK editor so users don't context-switch to a separate tool to see what changed

The biggest gap surfaced by this research

No iPaaS competitor offers the exact thing PMO-2980 describes: AI inserting connector code at the correct lines inside the platform's own editor, with a line-level before/after diff to accept or reject. Tray pushes that interaction out to Claude Code in your own IDE. Celigo keeps it in-app but as a staged-draft change summary, not an inline code diff. Boomi doesn't touch a code editor at all, it's a visual canvas. Fivetran with Claude in VS Code is the only one doing real in-editor insertion, and that lives in VS Code, not in Fivetran's own product. So the in-editor diff pattern is currently owned by the general coding tools (Cursor, Copilot), and they carry well-documented failure modes around destructive accept/reject (Theme 2) and wrong-line insertion (Theme 1). The opening for AIRO is to bring that interaction into a connector builder for the first time while designing around the failures the coding tools already shipped.

Part 4 — Theoretical framework: EDIT-Bench and the Shape of AI patterns

Source: EDIT-Bench: Evaluating LLM Abilities to Perform Real-World Instructed Code Edits, Chi, Chen, et al. (CMU, UC Berkeley), arXiv 2511.04486 (Nov 2025).

EDIT-Bench is the closest academic work to AIRO's feature. It's a 540-problem benchmark where the model has to use cursor position, highlighted code, and surrounding context (not just the instruction) to make the right edit. That is the edit-localization problem AIRO is taking on when it promises to update code at the correct lines. Its headline finding is that performance swings by up to 11% depending on how much contextual signal the model gets, and only one model clears 60%. In plain terms, getting the insertion point right is hard, and how much context you feed the model materially changes whether it lands.

How the framework maps to AIRO

EDIT-Bench concept AIRO equivalent
Cursor position and highlighted code as edit signal The SDK block and line range AIRO is asked to edit
Surrounding-context dependence (up to 11% swing) How much of the connector AIRO passes to the model when generating an edit
Edit-localization accuracy as the measured skill AIRO's "update at the correct lines" acceptance criterion

The practical takeaway is that the SDK block boundary AIRO already has is a strong localization signal, and using it deliberately as context could be worth more to insertion accuracy than a bigger model.

Shape of AI patterns worth borrowing

Two named patterns from the Shape of AI map onto this flow. Inline Action is targeting a specific span and showing the AI's change as a suggestion layer rather than an overwrite. Verification is requiring human approval before a change commits, and it explicitly names "loss of work from overwritten records" as the risk to design against, which is Theme 2 in this doc.

Constraints worth understanding

  • EDIT-Bench measures whether the model localizes and applies the edit correctly. It does not say anything about the review UI, so it informs the generation side, not the diff side.
  • The benchmark is on general code edits, not connector SDK code specifically, so the absolute accuracy numbers won't transfer directly.
  • The CSAP acceptance-prediction work (arXiv 2601.21379, Jan 2026) is adjacent but more about suppressing low-confidence suggestions than designing the diff, so it's a secondary reference at most.
  • None of these tell you how much SDK context is the right amount to pass. That's an empirical question worth testing on real connectors.
