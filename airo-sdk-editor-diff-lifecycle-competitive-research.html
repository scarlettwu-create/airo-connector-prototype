AIRO SDK editor: diff lifecycle and review controls (competitive research)

Context. Research for the AIRO SDK editor integration (PMO-2980), following the UX kickoff. The kickoff settled on a working model: AIRO writes into the editor as inline highlights, highlights accumulate across chat turns (diff baseline is the last saved version), an Accept/Keep control clears highlights without creating a version, every AI edit banks a restorable local draft checkpoint, and explicit save/test still creates numbered versions. This doc researches how the mature products handle exactly those mechanics and where they break. It's a companion to the earlier in-editor code insertion research, which covered insertion accuracy and destructive accept/reject; those findings are not repeated here.

Products covered. Cursor, GitHub Copilot in VS Code (agent mode / Edits), JetBrains AI Assistant + Junie, Windsurf (Cascade), Claude Code, Replit Agent.

Scope. Five mechanics of the in-editor diff lifecycle: how pending diffs accumulate across chat turns, what accept actually does and how it relates to versioning, per-edit restorable checkpoints, per-response change summaries, and navigating between changes in long files.

A note on Tier 1. The prior research already established that no direct iPaaS competitor (Tray, Celigo, Boomi, Fivetran in-product) does in-editor diffs at all, so this doc draws entirely from the coding tools where these mechanics actually exist. That selection skew is deliberate, not an oversight.

Part 1 — Diff lifecycle behavior per product

The six products split into three lifecycle models. Gate-before-write: the AI proposes, the user approves, then the file changes (Claude Code's default mode, Windsurf diff zones, JetBrains). Write-then-review: edits land in the file immediately and the user resolves pending review state afterward (Cursor, Copilot agent mode). Write-then-rollback: no review gate at all, control happens after the fact through checkpoints (Replit). The PMO-2980 kickoff model is write-then-review with checkpoint undo layered on, so Cursor and Copilot are the closest structural references and also carry the most documented failure history.

1.1 Cursor

Agent edits apply to files as it works, and since 2.0 (Oct 2025) a single review pane aggregates all pending changes across the conversation, with Cmd+Return to accept all and committing auto-accepting diffs per file since 2.1; checkpoints are created before significant modifications (not every edit), restore per request, and are explicitly not version control; each response ends with a change summary file list that links into the review pane since 2.3. No hunk-level keyboard navigation is documented, only file-level jumps in the review pane.

1.2 GitHub Copilot in VS Code

The agent applies and saves edits to disk first, then the user resolves them with Keep/Undo per file or for everything, and staging in Source Control auto-accepts pending edits, so accept is decoupled from saving but coupled to git staging; checkpoints are opt-in via a setting (snapshot before each request, restore also removes that request from the conversation, with a Redo to reapply); the editor overlay has Up/Down controls that step through edits and auto-reveal the next pending change when one is resolved. This is the most complete navigation story of the six.

1.3 JetBrains AI Assistant + Junie

Edit flows offer Accept All / Discard All per response with a per-change Revert button in the gutter, and Next/Previous Change buttons to step through proposed modifications; Junie adds a per-prompt "Rollback here" and a Done panel listing all changed files, all sitting on top of the IDE's Local History as a separate safety net. Cross-turn accumulation is not documented; review is per response.

1.4 Windsurf (Cascade)

Edits appear as per-hunk "diff zones" (inline highlighted regions with accept/reject controls on each hunk) that persist until resolved, and each prior prompt has a hover revert that rolls all code back to that step, with the docs warning "Reverts are currently irreversible, so be careful!". No per-response change summary or hunk navigation is documented. (Windsurf has since been folded into Cognition's Devin Desktop; docs now live there.)

1.5 Claude Code

Each edit is a gated proposal (side-by-side diff, approve before write), so there is no accumulated pending state by design; every user prompt creates a checkpoint, restorable via /rewind with separate options to restore code, conversation, or both, with the documented limitation that files changed by bash commands or external edits are not tracked. No structured per-response change summary UI; the VS Code extension opens diffs in the native diff viewer.

1.6 Replit Agent

No pre-accept gate at all: the agent writes directly and control is retroactive through checkpoints created at milestones (feature completed, before risky fixes), capturing files, conversation, environment, and optionally database, with bidirectional rollback; checkpoints generate git commits, making Replit the only product that ties checkpoints to real versions. Closest of the six to Workato's user profile (non-professional developers in a hosted editor), and notably it skipped the review gate entirely.

Part 2 — User pain points (themed insights)

Five themes, mined from forum.cursor.com, VS Code and Claude Code GitHub issues, the old Codeium/Windsurf tracker, Replit Discourse, and Hacker News. Two honesty notes. First, users complain about concrete breakage, not abstractions; "review fatigue" appears in research papers but not in user posts, so the themes below are named from what users actually hit. Second, Reddit and JetBrains YouTrack were not fetchable during this run, so JetBrains evidence is thin and noted as such.

Theme 1: Pending highlights turn into phantom state that won't clear

In both write-then-review products, the pending-diff state regularly detaches from reality: highlights survive accepting, buttons survive restarts, and the overlay claims manual edits as AI edits.

  • Copilot: "the leftover Keep or Undo button from previous session doesn't disappear which is confusing and I don't know which session generated which edits." (saadjs, vscode #295666, Feb 2026)
  • Copilot: "The pending edits diff overlay shows changes from the session's last checkpoint. Even if I manually edited those files outside the chat session... I cannot clear the diff overlay." (JoseCarrascoPujante, vscode #307439, Apr 2026)
  • Cursor: "at some point i had some pending changes from cursor that i accepted but the UI and green highlight persisted." (ArinCantCode, forum thread, Nov 2025)
  • Cursor: review controls appear "even when the file has no actual uncommitted or unreviewed changes" (user1085, forum thread, Nov 2025)
  • Cursor: "Review: All Changes shows the original changes that were made by the agent, disregarding the manual changes that the user did." (Al Gor, forum thread, Nov 2025)
  • Windsurf: renaming a file with pending diffs "breaks the visual diff display and acceptance functionality for all diffs... users lose the ability to see or accept any changes." (jeffcoughlin, codeium #114, Jan 2025)

The common root: these products track highlight state as a log of AI edit events, and the log drifts from the file whenever reality moves (a manual edit, a rename, a restart). The kickoff model defines the highlight as a comparison between two fixed states, current code vs last saved version, which is computed rather than remembered. For AIRO: that derived model is the single best protection against this whole theme, and it's worth defending it when implementation pressure pushes toward tracking individual edit events. It also cleanly answers "what happens when the user manually edits inside a highlight": recompute, and only real differences stay highlighted. The kickoff requirement that manual edits not be highlighted needs one refinement, though: if a manual edit changes a line relative to the last saved version, a pure two-state diff will highlight it anyway. Deciding whether that's acceptable (highlight means "changed since save") or not (highlight means "AIRO did this") is a real spec decision, because the two definitions diverge exactly when users mix manual and AI edits, which the workflow in the kickoff (prompt, review, refine by hand, test) guarantees they will.

Theme 2: Accept granularity is never right, and accept's relationship to versioning is fuzzy

Sub-pattern A: one accept unit doesn't fit review

Products that offer only coarse accept get asked for granular control, and products that offer only granular get asked for batch.

  • Claude Code: "users must currently accept or reject entire modifications without granular review capabilities." (Dhivyeshp, claude-code #44787, Apr 2026)
  • Claude Code: "With 'Ask before edits' I have to approve each individual line change without knowing what's coming next, I'm making decisions blind." (brilliant-directories user, claude-code #31888, Mar 2026)
  • Cursor: "I don't see the Keep All changes in the chat like i used to... i need to approve each file that got changed!" (rasheddev, forum thread, Aug 2025)
  • Windsurf: "The ability to accept or reject individual code changes made by Cascade has been removed... Changes are automatically applied with only a non-interactive diff view shown." (ff-will, codeium #131, Feb 2025)

For AIRO: the V1 single connector file makes this simpler than multi-file tools, and the kickoff's batch accept is the right starting unit. The complaints suggest the pair that works is batch accept plus per-hunk visibility (see Theme 5), with per-hunk accept as a possible fast-follow rather than a V1 requirement.

Sub-pattern B: nobody can say what accept commits

Users ask for accept to create a version, products quietly couple accept to git in one direction or another, and accepting sometimes does more than advertised.

  • Cursor: "for every accepted composer solution by 'Accept all', it will create a git commit with the description of the current composer chat." (feature request, Stavarc, forum thread, Nov 2024)
  • Copilot officially couples them: staging changes in Source Control auto-accepts pending edits; Cursor 2.1 made committing auto-accept diffs per file
  • Windsurf: "Sometimes the Accept changes button doesn't work... Sometimes it DELETES a lot of your code, without you noticing." (XCSme, HN, Jan 2025)
  • Windsurf: "When there are changes and I accept one of them, then trying to accept the others gives an error saying the file was changed." (sama004, HN, Nov 2025)

For AIRO: the kickoff already made the clean call (Accept clears highlights, save creates versions, the two are independent), and the competitor drift is in the opposite direction, toward coupling accept with commit. Both Copilot and Cursor auto-accept on commit because users treat saving as the stronger signal. Worth mirroring that one-way coupling: saving a version should clear highlights automatically (the kickoff says it does), and the accept control's only job is clearing highlights early. The label matters here; "Keep" describes clearing-without-versioning better than "Accept," which users read as "this is now committed."

Sub-pattern C: the review gate changes under users without warning

  • Cursor: "It has been so annoying that I do not get the explicit review of the files with highlighted changes which was one of the pros of using Cursor." (Aniket_Churihar, forum thread, Apr 2026)
  • Cursor: "Review and accepting/rejecting code is not a nice to have... I need this to work." (wadsworj, forum thread, Nov 2025)

The prior research flagged the same Cursor regression from the insertion angle. The signal repeats from this angle too: when the review surface disappears or silently changes granularity, users escalate immediately. For AIRO: treat the highlight-and-review surface as a stable contract once shipped.

Theme 3: Restore promises more than it keeps

Checkpoint restore is the trust backstop of the whole model, and in every product that ships it, restore does less than the button copy implies.

  • Cursor: "'Restore checkpoint' pop up saying 'you can always undo this later'. It seems you cannot undo this later." (Ursalbus, forum thread, Aug 2025)
  • Cursor: "it appears to restore the checkpoint as of when the question was completed, not when the question was started." (Charles, forum thread, Mar 2025)
  • Claude Code: "Use /rewind and select a checkpoint. There will be no option to restore the code, only the conversation." (hasen6, claude-code #15403, Dec 2025)
  • Claude Code: "When the response rolls back, those side effects persist but the user has no record of what was done." (martinogdenlabs, claude-code #29684, Feb 2026)
  • Replit: "I started a new chat and lost all of the rollback options." (mkennventure, Replit Discourse, Jan 2025)
  • Replit: "their database rollback is throwing errors." (ianmabie, HN, Nov 2025, after the agent truncated a production table)

Two distinct failure shapes. The first is scope: checkpoints cover less than users assume (Claude Code doesn't track bash side effects, Replit excludes the database by default, Cursor restores a different moment than expected). The second is lifetime: checkpoints vanish when the session changes. For AIRO: the SDK editor's single-file scope makes a checkpoint genuinely complete in a way these products can't offer, which is worth saying in the UI. The lifetime question is live, though: the kickoff's open item about Arrow chat history scoping (shared across asset types vs per connector) directly decides whether local draft checkpoints survive a closed tab or a new chat, and Replit's complaint shows what it feels like when they don't. Worth folding checkpoint lifetime into the Iliana alignment conversation, not treating it as an implementation detail.

Theme 4: The chat says one thing, the code says another

Across three products, the agent's own narration of what it changed routinely disagrees with the actual diff, in both directions.

  • Cursor: "it explicitly says that the code has been updated or fixed, but when I check the target file, no actual changes have been applied." (ehdgjs6203, forum thread, Mar 2026)
  • Copilot: "The edit summary shows +0 -0... Clicking into the diff view shows no changes... the file content is clearly modified." (cutepikachu-cn, vscode #303827, Mar 2026)
  • Copilot: "changes in file look like everything is changed, when in fact (git changes) only a few changes were made." (obiwanjacobi, vscode #297699, Feb 2026)
  • Replit: "The agent specifically told me it has fixed the issue, but again no changes were done." (Gcspratt, Replit Discourse, Oct 2025)
  • Replit: "coming back to me with an A-Okay message that it's fixed but nothing at all has changed." (rcrownholm, Replit Discourse, Aug 2025)

This lands directly on the kickoff's change-summary requirement, and it sharpens it: the danger isn't a missing summary, it's a summary generated by the model's narration rather than the actual edit. Model narration is what these users caught lying. For AIRO: worth specifying that the one-line summary is derived from the computed diff (which blocks changed, how many lines added or removed), with the model only phrasing it. A summary that can't disagree with the diff is the feature; a summary that can is a liability the competitors already demonstrate.

Theme 5: Finding the changes is the user's job

In long files, products under-invest in getting the reviewer to the changes, and the cost is unreviewed or accidentally accepted code.

  • Cursor: "There need to be keyboard shortcuts... that let us quickly navigate up and down the applied changes block by block... I find myself scrolling to find and audit the changes." (HTML, forum thread, Sep 2024)
  • Cursor: "it's easy to accidentally accept or reject off-screen changes." (dimitri-vs, same thread, Dec 2024)
  • Cursor: "I asked to change several lines. It did, but I can review only first one." (IAkumaI, forum thread, Feb 2026)
  • Cursor: "any deletions it makes will NOT show up marked in red when reviewing changes." (Marijn17s, forum thread, Nov 2025)
  • A CHI 2026 study of agent workflows recorded the same moment in the lab: participants lost track of what changed, with one reporting "I did not expect 10 files... 1000 lines each" (Chen et al.)

Copilot is the counter-example worth copying: Up/Down overlay controls plus auto-reveal of the next pending change on resolve, so the reviewer is always taken to the next decision (docs). For AIRO: with 1,000+ line connectors named in the kickoff, a jump-between-highlights control with a visible count ("3 changes") seems closer to must-have than polish, and dimitri-vs's complaint argues the accept-all control should communicate how many off-screen changes it covers. The Marijn17s quote also exposes a gap in a pure highlight model: deleted lines have nothing to highlight. The kickoff model (highlight = diff vs last save) needs an explicit answer for how a deletion is represented inline, because "the AI removed something and the editor shows no trace" is exactly the silent-edit failure the prior research warned about.

Part 3 — AIRO design implications

Competitor pain point What this could mean for AIRO
Pending highlights detach from file reality after manual edits, renames, restarts (Theme 1) Worth deriving highlight state by comparing current code to the last saved version on every render, never tracking it as a log of AI edit events
A two-state diff can't distinguish AI edits from manual edits to the same region (Theme 1) Worth deciding explicitly whether highlights mean "changed since save" or "AIRO did this," because the kickoff currently implies both
One accept unit never fits: per-line is blind, per-everything is reckless (Theme 2A) Worth pairing batch accept with per-hunk navigation and a visible change count in V1, deferring per-hunk accept
Users can't say what accept commits; Copilot and Cursor both auto-accept on commit (Theme 2B) Worth labeling the control "Keep" rather than "Accept," and keeping the one-way coupling: save clears highlights, but clearing highlights never saves
Review gates silently removed or changed cause immediate escalation (Theme 2C) Worth treating the highlight-review surface as a stable contract once shipped
Restore covers less than the button implies (Theme 3) Worth writing restore copy that states exactly what comes back (editor code, not test results or connection state)
Checkpoints vanish with the session (Theme 3) Worth folding checkpoint lifetime into the Arrow chat-history scoping alignment with Iliana, since chat scope decides draft scope
Model narration disagrees with the actual diff and users catch it (Theme 4) Worth deriving the one-line change summary from the computed diff, with the model only phrasing it
Off-screen changes get accepted unseen in long files (Theme 5) Worth a jump-between-highlights control plus a count on the accept control ("Keep 3 changes")
Deletions are invisible in highlight-only models (Theme 5) Worth specifying how a deleted line is represented inline before the highlight model is finalized

The biggest gap surfaced by this research

None of the six products ships a change summary that is guaranteed to match the code. Cursor's file list links to the review pane but says nothing about what changed; Claude Code and Windsurf have no structured summary at all; and where summaries exist, they are the model's narration, which users in three products caught reporting changes that never happened (Theme 4). The kickoff upgraded the one-line summary from good-to-have to should-have. The evidence here suggests going one step further in the PRD: specify the summary as a derived artifact of the computed diff (which SDK blocks changed, lines added and removed), phrased by the model but unable to contradict the editor. The NaturalEdit study below adds the second half: summaries help most when they're clickably linked to the code they describe, which in AIRO's case means the summary line in chat could double as the navigation entry point Theme 5 asks for. One feature, two validated needs, and no competitor has shipped it.

Part 4 — Theoretical framework: linked summaries and the review burden

Source: NaturalEdit: Code Modification through Direct Interaction with Adaptive Natural Language Representation, Tang et al., arXiv 2510.04494 (Oct 2025, revised Apr 2026).

NaturalEdit is the closest experimental evidence on the kickoff's change-summary question. It treats natural-language summaries as interactive representations bidirectionally linked to the code, and its user study (N=20) found developers validated AI changes faster and reported more confidence and control than with a diff alone. The operative detail is the linking: the summary works because clicking it takes you to the code it describes, not because the prose exists.

Supporting evidence on the review side

  • A field study of 13 developers plus a 99-person survey found experienced developers deliberately keep AI edits small and reviewable rather than batch-accepting, so review affordances are load-bearing for exactly the users AIRO's SDK builders resemble (Huang et al., Dec 2025)
  • A mining study of AI-generated pull requests found most receive no human review downstream, which makes the in-editor review moment the last realistic checkpoint before code ships (Duma et al., May 2026)
  • An analysis of 1,154 developer discussions names review burden as the central cost of AI codegen: the producer's speed gain is paid by whoever reviews (Baltes et al., Mar 2026)
  • The Shape of AI Footprints pattern (trace what the AI did from prompt to result) and Verification pattern are the industry names for the same two needs

How the framework maps to AIRO

Framework concept AIRO equivalent
Summary bidirectionally linked to code Chat summary line that scrolls the editor to the corresponding highlight
Adjustable abstraction (one line vs detail) One-line summary in chat, expandable to per-block detail
Validation speed as the measured outcome Time from AIRO edit to user clicking Keep or restoring

Constraints worth understanding

  • NaturalEdit studied general code editing with 20 participants, not connector SDK code or non-professional developers, so effect sizes won't transfer directly.
  • No paper runs the precise A/B the kickoff implies (diff alone vs diff plus one-line summary); NaturalEdit is adjacent evidence, not proof.
  • Academic work on how users mentally model AI-session checkpoints vs explicit versions appears not to exist yet (as of June 2026), so the local-draft-vs-numbered-version model will need validating with Workato's own users.
